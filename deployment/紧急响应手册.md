## 二、P0 — 后端服务宕机

**症状**：前端所有 API 返回 502/503，小程序扫码失败

### 排查（2分钟内定位根因）
```bash
# 1. 查看后端最后错误日志（最关键的一步，先看原因再操作）
tail -100 backend/logs/fashion-supplychain.log | grep -E "ERROR|FATAL|OOM|Exception" | head -20

# 2. 检查数据库是否在线
docker ps | grep fashion-mysql-simple
docker exec fashion-mysql-simple mysql -uroot -pchangeme fashion_supplychain -e "SELECT 1" 2>&1

# 3. 检查端口占用
lsof -i:8088 | head -5
```

### 快速重启（30秒恢复）
```bash
# 标准重启（推荐）
./dev-public.sh

# 如果 dev-public.sh 无效，手动重启
pkill -f "spring-boot" 2>/dev/null
sleep 2
cd backend
JAVA_HOME=/opt/homebrew/opt/openjdk@21/libexec/openjdk.jdk/Contents/Home \
  SPRING_DATASOURCE_URL=jdbc:mysql://localhost:3308/fashion_supplychain \
  SPRING_DATASOURCE_USERNAME=root \
  SPRING_DATASOURCE_PASSWORD=changeme \
  APP_AUTH_JWT_SECRET=ThisIsA_LocalJwtSecret_OnlyForDev_0123456789 \
```


## 四、P0 — 数据库迁移失败（服务无法启动）

**症状**：后端启动即崩溃，日志含 `FlywayException` 或 `Migration V*.sql failed`

### 排查
```bash
# 查看具体失败的迁移
tail -50 backend/logs/fashion-supplychain.log | grep -A 10 "FlywayException\|Migration"

# 查看 Flyway 执行历史
docker exec fashion-mysql-simple mysql -uroot -pchangeme fashion_supplychain \
  -e "SELECT version, description, state, installed_on \
      FROM flyway_schema_history ORDER BY installed_rank DESC LIMIT 10;"
```

### 修复方案A：脚本有 SQL 语法错误
```bash
# 1. 修复脚本 SQL 语法后，手动执行修复后的脚本
docker exec -i fashion-mysql-simple mysql -uroot -pchangeme fashion_supplychain \
  < backend/src/main/resources/db/migration/V失败版本号.sql

# 2. 将 Flyway 历史中的该版本标为已修复
docker exec fashion-mysql-simple mysql -uroot -pchangeme fashion_supplychain \
  -e "UPDATE flyway_schema_history SET state='SUCCESS' WHERE version='失败版本号';"

# 3. 重启后端，Flyway 检测到已修复后继续正常启动
./dev-public.sh
```

### 修复方案B：非幂等脚本重复执行（如 "Column already exists"）
```bash
# 跳过该版本（标记为已成功）
docker exec fashion-mysql-simple mysql -uroot -pchangeme fashion_supplychain \
  -e "UPDATE flyway_schema_history SET state='SUCCESS',checksum=-1 WHERE version='失败版本号';"

# 然后立即将该脚本改为幂等写法并修复后重推
```

### ⚠️ 预防原则（FLYWAY_ENABLED=true）
- `CREATE TABLE` → 必须 `CREATE TABLE IF NOT EXISTS`
- `ALTER TABLE ADD COLUMN` → 必须用 INFORMATION_SCHEMA 条件判断（MySQL 8.0 不支持 IF NOT EXISTS）
- 格式参考：
  ```sql
  SET @s = IF((SELECT COUNT(*) FROM INFORMATION_SCHEMA.COLUMNS
               WHERE TABLE_SCHEMA=DATABASE() AND TABLE_NAME='t_xxx' AND COLUMN_NAME='col')=0,
      'ALTER TABLE t_xxx ADD COLUMN col VARCHAR(64) NULL',
      'SELECT 1');
  PREPARE stmt FROM @s; EXECUTE stmt; DEALLOCATE PREPARE stmt;
  ```

## 五、P1 — 小程序扫码全部失败

**症状**：工厂反馈扫码无响应或持续报错

### 排查
```bash
# 1. 检查扫码接口连通性
curl -s http://localhost:8088/api/production/scan/execute \
  -X POST -H "Content-Type: application/json" \
  -d '{"test":true}' | head -20

# 2. 查看扫码相关日志（有 [ScanExec]/[BundleLookup]/[ScanSave] 诊断标签）
tail -200 backend/logs/fashion-supplychain.log | grep -E "\[ScanExec\]|\[BundleLookup\]|\[ScanSave\]|scan/execute"

# 3. 检查是否防重复误触发（菲号+工序维度的时间间隔）
docker exec fashion-mysql-simple mysql -uroot -pchangeme fashion_supplychain -e "
SELECT bundle_no, process_code, MAX(scan_time) as last_scan
FROM t_scan_record
WHERE scan_result='success' AND scan_time > NOW() - INTERVAL 2 HOUR
GROUP BY bundle_no, process_code ORDER BY last_scan DESC LIMIT 20;"
```

### 降级处理（无法立即修复时）
1. 通知工厂：**临时使用纸质记录** + 工序时间，系统恢复后由跟单员补录
2. 微信小程序「配置」→ 切换后端地址（如有备用地址）
3. 联系开发：在 `StageDetector.js#L610` 临时调大 `minInterval`（减少误报）

## 七、P1 — 工资结算数据异常

**症状**：工人工资金额明显偏高/偏低，或结算单卡在"对账中"

### 关键保护机制
- `ScanRecord.payrollSettled = true` → 撤回扫码会被系统拒绝（`ScanRecordOrchestrator` 第338/490行）
- 已通过审批的结算单无法直接删除（状态机保护）

### 排查
```bash
# 查看当前结算状态分布
docker exec fashion-mysql-simple mysql -uroot -pchangeme fashion_supplychain -e "
SELECT status, COUNT(*) FROM t_finished_product_settlement GROUP BY status;"

# 查看指定工人的结算明细
docker exec fashion-mysql-simple mysql -uroot -pchangeme fashion_supplychain -e "
SELECT operator_id, process_name, quantity, unit_price, amount, settled_at
FROM t_payroll_settlement
WHERE operator_id = 'WORKER_ID'
ORDER BY settled_at DESC LIMIT 20;"

# 查看是否有 payrollSettled=1 的扫码记录（不可撤回）
docker exec fashion-mysql-simple mysql -uroot -pchangeme fashion_supplychain -e "
SELECT id, bundle_no, quantity, payroll_settled, scan_time
FROM t_scan_record
WHERE order_id = ORDER_ID AND payroll_settled = 1 LIMIT 10;"
```

## 八、P1 — 进度球数值异常（偏高/偏低）

**症状**：订单进度球显示的件数明显不对

### 排查
```bash
# 1. 确认该订单的真实扫码记录
docker exec fashion-mysql-simple mysql -uroot -pchangeme fashion_supplychain -e "
SELECT progress_stage, process_name, SUM(quantity) as total_qty,
       COUNT(*) as scan_count, MAX(scan_time) as last_time
FROM t_scan_record
WHERE order_id = ORDER_ID AND scan_result='success'
GROUP BY progress_stage, process_name;"

# 2. 如果进度球数值与 DB 一致，问题在前端 useBoardStats.ts 的兜底逻辑
#    检查：是否有扫码记录但 hasScanByNode 没正确拦截兜底

# 3. 手动重新计算 productionProgress（确认后执行）
docker exec fashion-mysql-simple mysql -uroot -pchangeme fashion_supplychain -e "
UPDATE t_production_order
SET production_progress = (
  SELECT LEAST(100, ROUND(SUM(sr.quantity) * 100.0 / po.order_quantity))
  FROM t_scan_record sr
  WHERE sr.order_id = po.id AND sr.scan_result='success' AND sr.delete_flag=0
)
WHERE id = ORDER_ID AND status != 'COMPLETED';"
```

## 九、P2 — 数据库连接池告警（HikariPool Connection leak）

**症状**：日志持续输出 `Connection leak detection triggered`，API 响应变慢

### 临时处理
```bash
# 1. 识别慢操作类型
grep "Connection leak" backend/logs/fashion-supplychain.log | tail -20

# 2. 重启后端（临时清除泄漏连接，修改连接池参数后生效）
```

### 配置调整（`backend/src/main/resources/application.yml`）
```yaml
hikari:
  maximum-pool-size: 30          # 从20调高（并发高时使用）
  leak-detection-threshold: 60000  # 60秒（Excel导出等慢操作需要>30s）
```

## 十一、数据库备份与恢复

### 立即备份（遇到任何数据异常前先备份！）
```bash
# 备份到 deployment/backups/
docker exec fashion-mysql-simple mysqldump \
  -uroot -pchangeme --single-transaction fashion_supplychain \
  > deployment/backups/emergency_$(date +%Y%m%d_%H%M%S).sql

echo "备份完成：$(ls -lh deployment/backups/ | tail -1)"
```

### 恢复数据库
```bash
# ！！！恢复操作会覆盖当前所有数据，执行前务必再次备份当前数据！！！
docker exec -i fashion-mysql-simple mysql -uroot -pchangeme fashion_supplychain \
  < deployment/backups/your_backup_file.sql
```

### 查看最近备份
```bash
ls -lht deployment/backups/ | head -10
```

## 十二、回滚决策树

```
线上出现 bug
     ↓
影响全部用户（P0）？
     ├── 是 → 立即 git revert HEAD + push（3~5分钟自动重部署）
     │         推送前务必先备份数据库！
     └── 否 → 能在30分钟内热修复？
               ├── 是 → 修复 → 编译验证 → git add → push
               └── 否 → git revert + push，稳定后再排期重做
```

### git revert 命令（推荐，安全可追溯）
```bash
# 回滚最新一次提交
git revert HEAD --no-edit
git push upstream main
# 云端 3~5 分钟自动重部署

# 回滚到某个特定 commit（例如 23ce0529）
git revert 23ce0529 --no-edit
git push upstream main
```

### ❌ 禁止使用
```bash
git reset --hard  # 危险！会丢失提交历史，无法追溯
```
