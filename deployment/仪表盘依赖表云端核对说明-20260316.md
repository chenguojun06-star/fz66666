# 仪表盘依赖表云端核对说明

适用范围：当首页 dashboard 已排除 `t_production_order` 核心列问题，但仍怀疑 `GET /api/dashboard`、`GET /api/dashboard/quality-stats`、`GET /api/dashboard/top-stats`、`GET /api/dashboard/scan-count-chart`、`GET /api/dashboard/urgent-events` 因其他依赖表缺列而异常时使用。

## 需要执行的文件

1. 核对脚本：deployment/cloud-db-dashboard-dependent-tables-verify-20260316.sql

## 覆盖表

- `t_style_info`
- `t_scan_record`
- `t_material_purchase`
- `t_product_warehousing`

## 执行目标

这份脚本不修改任何数据，只回答两件事：

1. dashboard 依赖的其余核心表是否存在。
2. 这些表中 dashboard 实际会读取的核心列是否齐全。

## 执行方法

1. 打开微信云托管数据库控制台。
2. 直接执行 deployment/cloud-db-dashboard-dependent-tables-verify-20260316.sql。
3. 按结果判断：
   - `step-1` 出现 `MISSING_TABLE`：对应 dashboard 链路一定异常，先补表。
   - `step-2` 到 `step-5` 出现 `MISSING`：对应依赖表缺了 dashboard 实际读取的核心列，需要优先补列。
   - `step-6` 若所有 `missing_columns` 都为 `NULL`：说明这 4 张表的 dashboard 核心字段都齐全，后续应优先查云端部署版本、缓存或非 schema 类报错。

## 各表对应的 dashboard 依赖关系

### `t_style_info`

涉及功能：

- 首页最近动态里的“新增款号”
- 顶部统计中的“样衣开发数量”

本次核对核心列：

- `id`
- `style_no`
- `status`
- `create_time`
- `sample_status`

### `t_scan_record`

涉及功能：

- 首页最近动态里的“车间扫码”
- 智能日报里的今日扫码 / 近 7 天扫码
- 扫码次数 / 扫码数量折线图

本次核对核心列：

- `id`
- `order_no`
- `scan_time`
- `operator_name`
- `operator_id`
- `scan_result`
- `quantity`

### `t_material_purchase`

涉及功能：

- 首页最近动态里的“物料采购已创建”
- 紧急事件中的待处理采购统计

本次核对核心列：

- `id`
- `purchase_no`
- `status`
- `delete_flag`
- `create_time`

### `t_product_warehousing`

涉及功能：

- 智能日报里的昨日入库单数 / 件数
- 质检统计
- 顶部看板中的入库数量

本次核对核心列：

- `delete_flag`
- `create_time`
- `warehousing_end_time`
- `qualified_quantity`
- `unqualified_quantity`
- `defect_remark`

## 结果后的处理建议

1. 如果某张表缺的是 dashboard 核心列：优先补这张表，不要再回头盯 `t_production_order`。
2. 如果这 4 张表和 `t_production_order` 的 dashboard 核心列都齐全：首页再出 500 时，优先查云端是否部署到最新代码，或者直接查后端异常日志。
3. 如果后续又出现新的 schema drift，优先做两件事：热点查询显式 `.select(...)`，再补齐真正缺失的列。

## 补充说明

- 当前 dashboard 模块已经对 `ProductionOrder` 的主要高频读接口做了最小字段收敛；这份脚本用于把剩余非订单依赖表的 schema 风险也纳入核验。
- 当前 [cloudbaserc.json](../cloudbaserc.json) 中云端配置是 `FLYWAY_ENABLED=true`，这份脚本仍然只用于核验现状，不替代正式迁移。
