# 仪表盘订单表云端核对说明

适用范围：排查首页 `GET /api/dashboard/daily-brief` 与 `GET /api/dashboard/urgent-events` 返回 500，怀疑云端 `t_production_order` 缺列或迁移未执行的场景。

## 需要执行的文件

1. 核对脚本：deployment/cloud-db-production-order-dashboard-verify-20260316.sql

## 执行目标

这份脚本不修改任何数据，只回答三件事：

1. `t_production_order` 表是否存在。
2. 首页日报/紧急事件真正依赖的核心列是否齐全。
3. 订单实体里那些高风险扩展列是否缺失，后续是否还会拖垮其他订单接口。

## 执行方法

1. 打开微信云托管数据库控制台。
2. 直接执行 deployment/cloud-db-production-order-dashboard-verify-20260316.sql。
3. 按结果分组判断：
   - `step-2` 全部 `OK`：首页两个 dashboard 接口的核心列齐全，当前 500 更可能是旧代码未部署或其他表报错。
   - `step-2` 出现 `MISSING`：首页日报/紧急事件仍可能直接 500，优先补这些核心列。
   - `step-3` 出现 `MISSING`：即使首页修复后恢复，其他仍使用订单全字段查询的接口后续也可能继续报错。
   - `step-5` 某些 Flyway 版本 `success=0` 或无记录：优先排查为什么该迁移没成功落库。

## 本次重点核对的首页核心列

- `id`
- `order_no`
- `style_no`
- `factory_name`
- `production_progress`
- `planned_end_date`
- `status`
- `delete_flag`
- `create_time`

这些列是当前首页日报与紧急事件接口真正会读取的字段。只要这组列齐全，并且代码已升级为最小字段查询，首页接口就不该再被订单表无关扩展列拖成 500。

## 高风险扩展列说明

脚本还会核对以下高风险列：

- `progress_workflow_*`
- `skc`
- `factory_contact_person`
- `factory_contact_phone`
- `customer_id`
- `remarks`
- `expected_ship_date`
- `node_operations`
- `version`
- `order_biz_type`
- `org_unit_id` / `parent_org_unit_id` / `parent_org_unit_name` / `org_path` / `factory_type`
- `procurement_confirm_*`
- `urgency_level`
- `plate_type`
- `tenant_id`

这些列本次不是首页修复的直接依赖，但如果云端缺失，任何还在 `SELECT` 整个 `ProductionOrder` 实体的接口都有可能继续报数据库异常。

### 本次结果如何解读

- 如果 `missing_core_columns` 为 `NULL`：说明首页日报与紧急事件依赖的订单核心列齐全，当前 dashboard 500 不是由 `t_production_order` 核心列缺失直接造成。
- 如果 `missing_extension_columns` 只出现 `customer_id`：这代表 CRM 客户关联列尚未补齐，但它不是当前首页两个接口的直接依赖。
- 当前 [backend/src/main/java/com/fashion/supplychain/production/entity/ProductionOrder.java](../backend/src/main/java/com/fashion/supplychain/production/entity/ProductionOrder.java) 中 `customerId` 仍映射到 `company` 列，因此单独缺 `customer_id` 不应再把已经改成最小字段查询的 dashboard 接口拖成 500。
- 如果相关 Flyway 版本查询为空：只能说明这些补列迁移没有出现在当前库的 `flyway_schema_history` 里，不能单独据此判断首页故障；仍要结合核心列结果与云端部署版本一起看。

## 结果后的处理建议

1. 如果 `step-2` 缺列：先补首页核心列，再复测 dashboard。
2. 如果 `step-2` 全绿但 `step-3` 缺列：首页可先恢复，但应尽快补齐扩展列，避免其他订单接口继续出问题。
3. 如果 `step-5` 显示相关 Flyway 版本未成功：优先查部署日志和 `flyway_schema_history`，不要只盯着业务代码。
4. 如果 `step-2` 全绿且只缺 `customer_id`：优先确认最新 dashboard 修复代码是否已经部署到云端，再重测 `/api/dashboard/daily-brief` 与 `/api/dashboard/urgent-events`。

## 补充说明

- 当前 [cloudbaserc.json](../cloudbaserc.json) 中云端配置是 `FLYWAY_ENABLED=true`，所以这份脚本用于“核验现状”，不是替代 Flyway 正式迁移。
- 如果 verify 结果显示列缺失，而对应 Flyway 版本又不存在，说明云端部署时迁移没有真正落下去，需要继续查部署链路或补偿 SQL。
