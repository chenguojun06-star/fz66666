## 首次部署 / 重建环境

在云托管控制台 → backend 服务 → 服务设置 → 环境变量，配置以下变量：

### 数据库配置
- SPRING_DATASOURCE_URL：MySQL 内网连接地址
- SPRING_DATASOURCE_USERNAME：数据库用户名
- SPRING_DATASOURCE_PASSWORD：数据库密码

### Redis 缓存配置（Token 黑名单 + 会话）
- SPRING_REDIS_HOST：Redis 服务名（默认：`my-redis`）
- SPRING_REDIS_PORT：Redis 端口（默认：`6379`）
- SPRING_REDIS_PASSWORD：Redis 密码（务必设置强密码）

### 其他关键配置
- APP_AUTH_JWT_SECRET：JWT 密钥（32位以上）
- FLYWAY_ENABLED：**true**（cloudbaserc.json 已配置，Flyway 自动迁移数据库，无需手动执行 SQL）
- WECHAT_MP_APPID：小程序 AppID
- WECHAT_MP_SECRET：小程序 AppSecret
- WECHAT_TRUST_ALL_CERTS：**false**（微信云托管内网代理自签名证书场景设为 true，正常情况保持 false）

### WebSocket 实时通信
- 端点：`/ws/realtime`（已在 SecurityConfig 中 permitAll）
- 连接参数：`?userId=xxx&clientType=miniprogram|h5&tenantId=xxx`
- 心跳：客户端 18s + 服务端 20s（适配微信云托管 60s 空闲超时）
- 降级：WebSocket 重连 10 次失败后自动切换 30s 轮询

## Redis 缓存部署（可选，用于提升 Token 认证性能）

### 如何启用 Redis

1. **在微信云托管控制台新建 Redis 服务**
   - 镜像：`redis:7-alpine`
   - 服务名：`my-redis`（固定，backend 配置中硬编码）
   - Dockerfile：选「有」（走容器构建）
   - 启动参数：`--requirepass Fz2026Redis@8088`（示例密码，务必改成强密码）
   - 规格：CPU 0.25核 / 内存 0.5G（足够支持 5000+ 并发 Token 缓存）
   - 实例副本：1~5（自动扩缩容）

2. **配置 backend 环境变量**（见上方）
   - `SPRING_REDIS_HOST=my-redis`
   - `SPRING_REDIS_PASSWORD=你的强密码`

3. **验证连接**
   - backend 重部署后查看日志
   - 无 `connection refused` / `NOAUTH` 错误即成功

### Redis 对系统的影响

| 组件 | 作用 | 影响 |
|-----|------|------|
| Token 缓存 | JWT 黑名单 + 快速验证 | 认证延迟 <5ms，减少数据库查询 |
| 扫码业务 | **无直接依赖** | 扫码走 MySQL，Redis 宕机不影响扫码 |
| 并发能力 | 支持 1000~5000 人并发认证 | 无 Redis 时单机 200VU 上限 |

⚠️ **熔断保护**：`TokenAuthFilter` 内置 60 秒自动绕过逻辑，Redis 不可用时系统自动降级，不停服。

## 数据库迁移

✅ 云端 `FLYWAY_ENABLED=true`，Flyway **自动执行**迁移脚本，push 到 main 后 3~5 分钟容器重启时自动完成。

正确做法：

1. 在 `backend/src/main/resources/db/migration/` 编写 `V*.sql`（用于版本留痕）。
2. 所有 `ADD COLUMN` 必须用 INFORMATION_SCHEMA 幂等写法（禁止直接 `ALTER TABLE`，否则重复执行报错）。
3. push 到 main → CI/CD 自动部署 → Flyway 自动执行 → 日志确认无报错即完成。

⚠️ **Flyway 铁则（违反 = 全系统 500）**：

- **禁止修改已执行过的 V*.sql 文件内容**（checksum 不匹配 → Flyway 拒绝启动 → 全部 API 500）
- 已执行脚本发现问题 → 创建**新版本号脚本**来补偿（如 `V20260422001__fix_xxx.sql`）
- 新脚本必须本地 `mvn clean compile` 通过后再 push

冒烟校验（发布后）：

- 查看云端容器日志确认 `Successfully applied N migration(s)` 或 `Schema is up to date`
- 关键接口冒烟（下单/扫码/入库）确认无 500

## ⚠️ 重要：防止容器休眠导致的接口报错

### 问题现象
日志反复出现：`Thread starvation or clock leap detected (housekeeper delta=15m...)`
首个请求/唤醒后第一次操作偶发报错（数据库连接失效）

### 根本原因
微信云托管在无流量时会自动**冻结容器**（约16分钟），JVM 被暂停后：
1. HikariCP 内的空闲连接被 MySQL 断开（变成死连接）
2. 容器唤醒后第一个请求用到死连接 → 报错
3. 这就是为什么本地开发好好的，上了云就出问题

### 解决方式（二选一，推荐同时做）

**方式1：控制台设置最小实例数为 1（最根本）**
> 微信云托管控制台 → backend 服务 → 服务设置 → 扩缩容配置 → **最小实例数设为 1**
> 
> 这样容器永不休眠，无需等待冷启动，接口响应稳定。
> 注意：会产生少量费用（按实例运行时间计费）

**方式2：HikariCP 配置已优化（已提交）**
> `application-prod.yml` 已将 `minimum-idle` 改为 `0`，添加了 `keepalive-time: 120000`
> 
> 容器唤醒后 HikariCP 会自动创建新连接，不再使用死连接  
> 但首个请求仍有约 1~3 秒建连延迟（无法完全消除）
