### 1. StyleInfoController（23 → 10端点，-13）

**问题**：14个状态转换端点分散在5个阶段（pattern/sample/bom/process/secondary），每个阶段有start/complete/reset等操作。

**解决方案**：统一端点 `POST /{id}/stage-action?stage=xxx&action=xxx`

#### 新端点设计

```java
POST /api/style/info/{id}/stage-action
Query参数：
  - stage: pattern/sample/bom/process/secondary
  - action: start/complete/reset/progress/skip
Body参数（可选）：
  - 用于reset和progress操作
```

#### 智能路由逻辑

```java
@PostMapping("/{id}/stage-action")
public Result<?> stageAction(
        @PathVariable Long id,
        @RequestParam String stage,
        @RequestParam String action,
        @RequestBody(required = false) Map<String, Object> body) {
    
    switch (stage.toLowerCase()) {
        case "pattern":
            switch (action.toLowerCase()) {
                case "start": return Result.success(styleInfoOrchestrator.startPattern(id));
                case "complete": return Result.success(styleInfoOrchestrator.completePattern(id));
                case "reset": return Result.success(styleInfoOrchestrator.resetPattern(id, body));
                // ...
            }
        case "sample":
            // 4个操作：start/progress/complete/reset
        case "bom":
            // 2个操作：start/complete
        case "process":
            // 2个操作：start/complete
        case "secondary":
            // 3个操作：start/complete/skip
    }
}
```

#### 废弃端点（14个，3个月兼容）


### 2. UserController（14 → 12端点，-2）

**问题**：审批相关端点分散（/pending, /approve, /reject）。

**解决方案**：统一审批端点 `POST /{id}/approval-action?action=xxx`

#### 新端点设计

```java
POST /api/system/user/{id}/approval-action
Query参数：
  - action: approve/reject
Body参数（可选）：
  - approvalRemark: 审批意见
  - operationRemark: 操作备注
```

#### 智能路由逻辑

```java
@PostMapping("/{id}/approval-action")
public Result<?> approvalAction(
        @PathVariable Long id,
        @RequestParam String action,
        @RequestBody(required = false) User body) {
    
    String remark = body == null ? null : body.getApprovalRemark();
    if (remark == null) {
        remark = body == null ? null : body.getOperationRemark();
    }
    
    switch (action.toLowerCase()) {
        case "approve":
            userOrchestrator.approveUser(id, remark);
            return Result.successMessage("用户已批准");
        case "reject":
            userOrchestrator.rejectUser(id, remark);
            return Result.successMessage("用户已拒绝");
        default:
            return Result.fail("不支持的操作: " + action);
    }
}
```

#### 废弃端点（2个）

```java
@Deprecated // 将在 2026-05-01 移除
```

### 3. PatternProductionController（12 → 9端点，-3）

**问题**：重复的操作端点（/{id}/receive 和 /{patternId}/receive 都存在）和分散的工作流端点。

**解决方案**：统一工作流端点 `POST /{id}/workflow-action?action=xxx`

#### 新端点设计

```java
POST /api/production/pattern/{id}/workflow-action
Query参数：
  - action: receive/complete/warehouse-in/maintenance
Body参数（可选）：
  - remark: 备注（用于warehouse-in）
  - reason: 原因（用于maintenance）
  - releaseTime: 下板时间（用于receive）
  - deliveryTime: 交板时间（用于receive）
```

#### 智能路由逻辑

```java
@PostMapping("/{id}/workflow-action")
public Result<?> workflowAction(
        @PathVariable String id,
        @RequestParam String action,
        @RequestBody(required = false) Map<String, Object> request) {
    
    switch (action.toLowerCase()) {
        case "receive":
            return receive(id, request);
        case "complete":
            Map<String, Object> completeRequest = new HashMap<>();
            completeRequest.put("patternId", id);
            completeRequest.put("operationType", "COMPLETE");
            completeRequest.put("operatorRole", "PLATE_WORKER");
            return submitScan(completeRequest);
        case "warehouse-in":
            return warehouseIn(id, request);
        case "maintenance":
            // ...
        default:
            return Result.fail("不支持的操作: " + action);
    }
}
```

#### 废弃端点（4个，含重复端点）

```java
@Deprecated // 将在 2026-05-01 移除
```

## 🚀 技术实现亮点

### 1. 状态机模式（StyleInfoController）

**核心思想**：用两个维度（stage + action）替代14个独立端点

```
阶段（stage）× 操作（action）= 状态转换矩阵

pattern    → [start, complete, reset]        = 3个端点
sample     → [start, progress, complete, reset] = 4个端点
bom        → [start, complete]               = 2个端点
process    → [start, complete]               = 2个端点
secondary  → [start, complete, skip]         = 3个端点
──────────────────────────────────────────────────
14个分散端点 → 1个统一端点（stage-action）
```

**优势**：
- ✅ 易扩展：新增阶段只需在switch中添加case
- ✅ 易维护：所有状态转换逻辑集中管理
- ✅ 易测试：单一端点测试覆盖所有场景
- ✅ 前端友好：统一API调用模式

### 2. 审批流程模式（UserController）

**核心思想**：用action参数区分审批操作（approve/reject）

```
操作（action）→ 审批结果

approve → userOrchestrator.approveUser(id, remark)
reject  → userOrchestrator.rejectUser(id, remark)
```

**优势**：
- ✅ 逻辑内聚：审批相关操作统一管理
- ✅ 参数复用：remark字段支持多种来源（approvalRemark, operationRemark）
- ✅ 易扩展：可添加更多审批操作（如suspend、cancel）

### 3. 工作流模式（PatternProductionController）

**核心思想**：用action参数区分工作流阶段（receive/complete/warehouse-in/maintenance）

```
操作（action）→ 工作流节点

receive      → receive(id, request)
complete     → submitScan(COMPLETE)
warehouse-in → warehouseIn(id, request)
maintenance  → maintenance(id, reason)
```

**优势**：
- ✅ 消除重复：合并 /{id}/receive 和 /{patternId}/receive
- ✅ 流程清晰：工作流各阶段统一管理
- ✅ 参数灵活：不同操作支持不同参数
