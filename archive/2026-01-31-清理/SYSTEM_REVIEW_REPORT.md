# 全面架构审查与功能完整性核查报告

## 1. 概览 (Executive Summary)
本报告针对“服装66666”系统的 **样衣仓库**、**面辅料进销存**、**成品进销存** 三大核心模块进行了深度审查。
审查结果显示：系统后端架构基础良好（Spring Boot + MyBatis Plus），但**业务闭环存在严重缺失**。目前的“仓库系统”本质上是“单据记录系统”，缺乏核心的“实时库存（Stock Balance）”管理能力，导致账实无法自动同步。

---

## 2. 系统架构审查 (System Architecture Review)

### 2.1 整体架构
- **优点**: 采用标准的分层架构 (Controller/Service/DAO)，模块划分清晰。
- **缺点**: **领域边界模糊**。库存逻辑分散在 `Purchase` (采购) 和 `Warehousing` (入库) 服务中，缺乏统一的 `InventoryDomainService` (库存领域服务) 来集中管理库存变动。

### 2.2 数据库设计
- **完整性**: 
  - ✅ 基础单据表 (`t_material_purchase`, `t_product_warehousing`) 齐全。
  - ❌ **严重缺失**: 缺少 `t_sample_stock` (样衣库存表) 和 `t_material_stock` (面辅料库存表)。
  - ⚠️ **性能隐患**: 库存查询目前依赖“实时聚合”（如 `sumOutstockByOrderId`），随着数据量增长，计算性能会急剧下降。应引入“快照余量”字段。

### 2.3 API 设计
- **规范性**: 遵循 RESTful 风格，但部分接口职责过重（如 `saveAndValidate` 混合了业务校验与持久化）。

---

## 3. 功能模块完整性核查 (Functional Integrity Check)

### 3.1 样衣仓库 (Sample Warehouse)
> **状态: 严重缺失**
- **现状**: 仅有样衣资料 (`StyleInfo`) 和 BOM (`StyleBom`)。
- **缺失**:
  - 无法追踪样衣的去向（如：在展厅、借出给客户、设计室修改中）。
  - 缺乏“样衣借还”流程。
  - 缺乏样衣唯一码（RFID/二维码）管理。

### 3.2 面辅料进销存 (Fabric/Accessory Inventory)
> **状态: 逻辑断层**
- **现状**: 
  - 采购入库 (`MaterialPurchase`) 仅记录了单据，**未增加库存**。
  - 生产领料没有系统记录，无法计算“理论耗用”。
- **缺失**:
  - **实时库存**: 没有表记录当前某种面料剩多少米。
  - **领料/退料**: 缺乏生产领料单功能。
  - **自动扣减**: 生产完成时未自动扣减面料库存。

### 3.3 成品进销存 (Finished Goods Inventory)
> **状态: 半成品**
- **现状**: 
  - 入库 (`ProductWarehousing`) 已实现库存自动增加（由近期修复完成）。
  - **出库 (`ProductOutstock`) 尚未对接库存扣减**。目前出库仅校验了“入库总数”，未扣减 `t_product_sku` 中的实时库存。
- **风险**: 存在超卖风险，且无法支持电商平台的实时库存同步。

---

## 4. 维护性与代码质量 (Maintainability & Code Quality)

- **硬编码**: 状态值（如 "pending", "completed"）散落在各个 Service 中，未统一枚举管理。
- **异常处理**: 部分数据库操作未捕获特定异常，可能导致前端收到通用的 500 错误而非具体的业务提示。
- **测试覆盖**: 核心库存逻辑缺乏单元测试覆盖，容易在后续迭代中被破坏。

---

## 5. 改进建议与实施计划 (Improvement Plan)

### 第一阶段：核心补全 (立即执行)
> 目标：打通“进-销-存”的数据闭环，确保有库存可查。

1.  **成品出库闭环**:
    - 修改 `ProductOutstockServiceImpl`，在保存出库单后，调用 `ProductSkuService.updateStock` 扣减库存。
2.  **面辅料库存化**:
    - 创建 `t_material_stock` 表 (字段: `material_id`, `quantity`, `warehouse_location`)。
    - 修改 `MaterialPurchaseServiceImpl`，在“确认收货”时增加库存。
3.  **样衣管家**:
    - 创建 `t_sample_stock` 表，管理样衣的物理位置和状态。

### 第二阶段：业务深化 (2周内)
> 目标：完善业务流程，增加校验。

1.  **生产领料**: 开发“领料单”功能，生产开工时冻结/扣减面辅料库存。
2.  **样衣借还**: 开发样衣借用、归还、盘点流程。

### 第三阶段：性能与扩展 (1个月内)
1.  **库存预警**: 增加库存上下限报警。
2.  **报表优化**: 基于每日库存快照生成进销存月报。
