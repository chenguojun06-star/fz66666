# 扫码工序识别错误修复方案

## 📋 问题诊断

### 问题现象
订单 `PO20260125003` 扫码流程：
1. 第1次手动扫码 → 应该是"车缝"
2. 实际识别为 → **"大烫"** ❌（直接跳过了车缝）

### 根本原因

**扫码历史统计错误**，包含了系统自动生成的记录：

```
扫码记录：
1. "下单" (ORDER_CREATED:xxx)     ← 系统自动生成
2. "裁剪" (CUTTING_BUNDLED:xxx)   ← 系统自动生成
3. "大烫" (手动扫码)              ← 第1次手动扫码，但被当作第3次
```

**代码逻辑**：
```javascript
const scanHistory = await this._getScanHistory(orderNo, bundleNo);
const scanCount = scanHistory.length;  // scanCount = 3（包含了系统记录）

const sewingProcessList = ["车缝"];    // 只有1个车缝工序

if (scanCount < sewingProcessList.length) {  // if (3 < 1) → false
  return sewingProcessList[scanCount];
}
return '大烫';  // ← 错误！实际只扫了1次，但被认为已完成3次
```

### 业务上的混淆

**你问得对！** 业务上1个工序和多个工序**确实没有区别**：
- 1个车缝工序：扫1次 → 完成
- 5个车缝工序：扫5次 → 完成

**但代码实现有Bug**：
- 统计扫码次数时，**错误地把系统自动生成的记录也算进去了**
- 导致"第1次手动扫码"被当作"第3次扫码"
- 因为 3 > 1，所以认为车缝已完成，返回"大烫"

## ✅ 完整解决方案

### 核心修复：过滤系统自动生成的记录

**文件**：`miniprogram/pages/scan/services/StageDetector.js`

**修改位置**：`_getScanHistory()` 方法

```javascript
async _getScanHistory(orderNo, bundleNo) {
  const historyRes = await this.api.production.myScanHistory({...});
  const allRecords = historyRes && historyRes.records ? historyRes.records : [];

  // ✅ 修复：过滤掉系统自动生成的记录
  const manualRecords = allRecords.filter(record => {
    const requestId = (record.requestId || '').trim();
    // 排除系统自动生成的记录
    const isSystemGenerated =
      requestId.startsWith('ORDER_CREATED:') ||
      requestId.startsWith('CUTTING_BUNDLED:') ||
      requestId.startsWith('SYSTEM:');
    
    return !isSystemGenerated;
  });

  return manualRecords;  // 只返回手动扫码记录
}
```

**效果**：
- ✅ 系统记录不计入扫码次数
- ✅ 第1次手动扫码 → scanCount = 0 → 返回 sewingProcessList[0] = "车缝"
- ✅ 第2次手动扫码 → scanCount = 1 → 返回"大烫"

## 🎯 工序数量不固定的支持

### Q：如果订单有5个车缝工序，有的有8个，会有影响吗？

**A：完全没有影响！** ✅

代码逻辑是**动态筛选**的：

```javascript
// 步骤1：从配置中筛选出所有车缝工序
const sewingProcesses = nodes
  .filter(node => node.progressStage === '车缝')  // 动态筛选
  .sort((a, b) => a.sortOrder - b.sortOrder)      // 按顺序排序
  .map(node => node.name);                        // 提取名称

// 步骤2：根据扫码次数匹配
if (scanCount < sewingProcesses.length) {
  return sewingProcesses[scanCount];  // 返回第 N 个工序
}
return '大烫';  // 所有工序完成
```

**示例**：

| 订单配置 | 工序列表 | 扫码逻辑 |
|---------|---------|---------|
| 1个工序 | `['车缝']` | 扫1次 → 大烫 |
| 5个工序 | `['做领','上领','肩缝','袖缝','侧缝']` | 扫5次 → 大烫 |
| 8个工序 | `['裁片','锁边','做领','上领',...]` | 扫8次 → 大烫 |

**结论**：
- ✅ 支持任意数量的车缝工序（1 ~ N个）
- ✅ 自动适配扫码次数
- ✅ 每个订单可以有不同的工序配置
