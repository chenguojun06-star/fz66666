# 服装供应链管理系统 - 双端同步问题报告

**报告日期：** 2026-01-31  
**报告版本：** v1.0  
**检查范围：** 小程序与PC端数据同步机制  
**检查人员：** 架构评估团队  

---

## 执行摘要

本次检查对服装供应链管理系统的小程序端与PC端之间的数据同步机制进行了全面审查。发现**4个高风险问题**、**4个中风险问题**和**3个低风险问题**。主要问题集中在缺乏实时通信机制、token管理不完善、缓存策略简单等方面。

### 关键发现

| 风险等级 | 问题数量 | 主要问题 |
|---------|---------|---------|
| 🔴 高风险 | 4个 | 无WebSocket、token无刷新、多端冲突、无请求去重 |
| 🟡 中风险 | 4个 | 轮询策略固定、深比较性能差、无自动重连、无版本控制 |
| 🟢 低风险 | 3个 | 明文存储、事件不持久化、无缓存限制 |

**综合评估：** 当前同步机制基本可用，但实时性和可靠性有待提升。

## 一、问题详细分析

### 🔴 高风险问题

#### 问题1：无WebSocket实时通信机制

**问题描述：**
系统完全依赖HTTP轮询（30秒间隔）实现数据同步，没有使用WebSocket、SSE或其他实时通信技术。

**影响范围：**
- 扫码数据同步延迟最高30秒
- 订单状态变更无法实时通知
- PC端和小程序端数据不同步

**触发条件：**
- 用户在PC端操作后，小程序端需要等待下次轮询才能看到更新
- 网络延迟时，轮询间隔可能延长

**代码位置：**
```javascript
// syncManager.js
startSync(taskId, fetchFn, interval = 30000) {
  const timer = setInterval(() => {
    this._executeSync(config);
  }, config.interval); // 固定30秒轮询
}
```

**改进建议：**
```javascript
// 引入WebSocket
class RealtimeSync {
  constructor() {
    this.ws = null;
    this.reconnectAttempts = 0;
  }
  
  connect() {
    this.ws = wx.connectSocket({
      url: 'wss://api.example.com/ws'
    });
    
    this.ws.onMessage((res) => {
      const data = JSON.parse(res.data);
      this.handleRealtimeUpdate(data);
    });
  }
  
  handleRealtimeUpdate(data) {
    // 实时更新本地数据
    eventBus.emit(data.type, data.payload);
  }
}
```

**优先级：** P0（最高）  
**预计修复时间：** 2-3周  
**业务影响：** 高（影响用户体验）

#### 问题2：Token无自动刷新机制

**问题描述：**
当token过期时，系统直接跳转登录页，没有尝试静默刷新token，导致用户操作中断。

**影响范围：**
- 用户正在操作时突然跳转到登录页
- 表单数据可能丢失
- 用户体验差

**触发条件：**
- Token过期时间到达（默认2小时）
- 后端强制刷新token

**代码位置：**
```javascript
// request.js
function handle401Error(statusCode, body, skipAuthRedirect, reject) {
  clearToken();
  if (!skipAuthRedirect) {
    triggerLoginRedirect(); // 直接跳转，无刷新尝试
    reject(createError('未登录', { type: 'auth', statusCode, data: body }));
    return true;
  }
}
```

**改进建议：**
```javascript
// request.js
async function handle401WithRefresh(error) {
  try {
    const refreshToken = getRefreshToken();
    if (!refreshToken) {
      throw new Error('No refresh token');
    }
    
    const resp = await api.auth.refreshToken({ refreshToken });
    setToken(resp.data.token);
    setRefreshToken(resp.data.refreshToken);
    
    // 重试原请求
    return retryRequest(error.config);
  } catch (e) {
    // 刷新失败，跳转登录
    clearToken();
    redirectToLogin();
  }
}
```

**优先级：** P0（最高）  
**预计修复时间：** 1周  
**业务影响：** 高（影响用户体验）

#### 问题3：多端Token冲突

**问题描述：**
小程序和PC端使用相同的token机制，一端登录可能导致另一端token失效。

**影响范围：**
- 用户在PC端登录后，小程序端被登出
- 反之亦然
- 无法实现多端同时在线

**触发条件：**
- 后端采用单点登录策略
- 新登录会吊销旧token

**代码位置：**
```javascript
// storage.js
const TOKEN_KEY = 'auth_token'; // 小程序和PC使用相同的key

// PC端和小程序端都存储在同一个key下
function setToken(token) {
  wx.setStorageSync(TOKEN_KEY, token);
}
```

**改进建议：**
```javascript
// 方案1：使用独立token
const MP_TOKEN_KEY = 'mp_auth_token';
const PC_TOKEN_KEY = 'pc_auth_token';

// 方案2：后端支持多设备登录
// 后端返回设备标识，不同设备token独立
{
  "token": "xxx",
  "deviceId": "mp_xxx",
  "supportsMultiDevice": true
}
```

**优先级：** P1（高）  
**预计修复时间：** 1-2周（需后端配合）  
**业务影响：** 中（影响多设备用户）

#### 问题4：无请求去重机制

**问题描述：**
快速重复点击或网络抖动可能导致相同请求多次发送，造成数据异常（如重复扫码）。

**影响范围：**
- 重复扫码记录
- 重复提交订单
- 数据不一致

**触发条件：**
- 用户快速双击按钮
- 网络延迟时用户重复点击
- 请求超时后自动重试与手动重试冲突

**代码位置：**
```javascript
// scan/index.js
async onScan() {
  // 无去重保护
  const result = await api.production.executeScan(payload);
}
```

**改进建议：**
```javascript
// request.js
const pendingRequests = new Map();

function request(options) {
  const key = `${options.url}_${JSON.stringify(options.data)}`;
  
  // 检查是否有相同请求正在进行
  if (pendingRequests.has(key)) {
    return pendingRequests.get(key);
  }
  
  const promise = doRequest(options).finally(() => {
    pendingRequests.delete(key);
  });
  
  pendingRequests.set(key, promise);
  return promise;
}

// 或者使用防抖
const debouncedScan = debounce(onScan, 300, { leading: true, trailing: false });
```

**优先级：** P1（高）  
**预计修复时间：** 3-5天  
**业务影响：** 中（可能导致数据错误）

### 🟡 中风险问题

#### 问题5：轮询频率固定，无动态调整

**问题描述：**
数据同步使用固定30秒轮询，无论页面是否可见、网络状况如何，都保持相同频率。

**影响范围：**
- 不必要的网络开销
- 电池消耗增加
- 服务器压力增大

**触发条件：**
- 页面在后台运行
- 网络状况差
- 数据更新频率低的时间段

**改进建议：**
```javascript
// syncManager.js
class AdaptiveSyncManager {
  constructor() {
    this.baseInterval = 30000;
    this.currentInterval = 30000;
    this.pageVisible = true;
    this.networkQuality = 'good';
  }
  
  adjustInterval() {
    if (!this.pageVisible) {
      this.currentInterval = 300000; // 后台5分钟
    } else if (this.networkQuality === 'poor') {
      this.currentInterval = 60000; // 网络差1分钟
    } else {
      this.currentInterval = this.baseInterval;
    }
  }
  
  onPageVisibilityChange(visible) {
    this.pageVisible = visible;
    this.adjustInterval();
  }
}
```

**优先级：** P2（中）  
**预计修复时间：** 1周  
**业务影响：** 低（性能和资源优化）

#### 问题6：JSON.stringify深比较性能差

**问题描述：**
同步管理器使用JSON.stringify对比数据变化，大数据量时性能较差。

**影响范围：**
- 大数据量列表同步时卡顿
- 内存占用增加
- 对比时间随数据量线性增长

**代码位置：**
```javascript
// syncManager.js
_defaultCompare(newData, oldData) {
  return JSON.stringify(newData) !== JSON.stringify(oldData);
}
```

**改进建议：**
```javascript
// 使用更高效的差异检测
function shallowCompare(newData, oldData) {
  if (newData === oldData) return false;
  if (!newData || !oldData) return true;
  
  const newKeys = Object.keys(newData);
  const oldKeys = Object.keys(oldData);
  
  if (newKeys.length !== oldKeys.length) return true;
  
  return newKeys.some(key => newData[key] !== oldData[key]);
}

// 或者使用版本号机制
function versionCompare(newData, oldData) {
  return newData._version !== oldData._version;
}
```

**优先级：** P2（中）  
**预计修复时间：** 3-5天  
**业务影响：** 低（性能优化）

#### 问题7：无网络恢复自动重连

**问题描述：**
当网络断开恢复后，系统不会自动恢复数据同步，需要用户手动刷新。

**影响范围：**
- 网络恢复后数据不同步
- 用户需要手动刷新页面
- 可能错过重要数据更新

**改进建议：**
```javascript
// app.js
App({
  onLaunch() {
    // 监听网络状态变化
    wx.onNetworkStatusChange((res) => {
      if (res.isConnected) {
        // 网络恢复，重新启动同步
        this.resumeSync();
      } else {
        // 网络断开，暂停同步
        this.pauseSync();
      }
    });
  },
  
  resumeSync() {
    // 刷新所有同步任务
    syncManager.resumeAll();
    // 触发全局刷新
    eventBus.emit('NETWORK_RESTORED');
  }
});
```

**优先级：** P2（中）  
**预计修复时间：** 3-5天  
**业务影响：** 中（影响离线场景）

#### 问题8：缓存无版本控制

**问题描述：**
本地缓存没有版本号控制，当数据结构变更时，旧格式数据可能导致错误。

**影响范围：**
- 应用更新后数据解析错误
- 可能导致白屏或功能异常
- 需要用户手动清理缓存

**改进建议：**
```javascript
// storage.js
const DATA_VERSION = 'v1.0';
const VERSION_KEY = 'data_version';

function setStorageWithVersion(key, value) {
  const data = {
    _version: DATA_VERSION,
    _timestamp: Date.now(),
    value
  };
  wx.setStorageSync(key, data);
}

function getStorageWithVersion(key, fallback) {
  const data = wx.getStorageSync(key);
  
  if (!data || data._version !== DATA_VERSION) {
    // 版本不匹配，清理并返回默认值
    wx.removeStorageSync(key);
    return fallback;
  }
  
  return data.value;
}

// 应用启动时检查版本
function checkDataVersion() {
  const currentVersion = wx.getStorageSync(VERSION_KEY);
  if (currentVersion !== DATA_VERSION) {
    // 版本升级，清理所有缓存
    clearAllStorage();
    wx.setStorageSync(VERSION_KEY, DATA_VERSION);
  }
}
```

**优先级：** P2（中）  
**预计修复时间：** 1周  
**业务影响：** 中（影响应用更新）

### 🟢 低风险问题

#### 问题9：敏感数据明文存储

**问题描述：**
Token等敏感数据使用明文存储在本地，存在安全风险。

**改进建议：**
```javascript
// storage.js
const CryptoJS = require('crypto-js');
const SECRET_KEY = 'your-secret-key'; // 应从安全渠道获取

function encrypt(data) {
  return CryptoJS.AES.encrypt(JSON.stringify(data), SECRET_KEY).toString();
}

function decrypt(encrypted) {
  const bytes = CryptoJS.AES.decrypt(encrypted, SECRET_KEY);
  return JSON.parse(bytes.toString(CryptoJS.enc.Utf8));
}

function setSecureToken(token) {
  const encrypted = encrypt(token);
  wx.setStorageSync(TOKEN_KEY, encrypted);
}

function getSecureToken() {
  const encrypted = wx.getStorageSync(TOKEN_KEY);
  return encrypted ? decrypt(encrypted) : '';
}
```

**优先级：** P3（低）  
**预计修复时间：** 3-5天  
**业务影响：** 低（安全加固）

#### 问题10：事件只在运行时有效

**问题描述：**
事件总线只在应用运行时有效，页面重启后事件丢失，无法恢复。

**改进建议：**
```javascript
// eventBus.js
class PersistentEventBus extends EventBus {
  constructor() {
    super();
    this.persistentEvents = new Set(['ORDER_UPDATED', 'SCAN_SUCCESS']);
    this.loadPersistentEvents();
  }
  
  emit(eventName, data) {
    super.emit(eventName, data);
    
    if (this.persistentEvents.has(eventName)) {
      // 持久化存储
      wx.setStorageSync(`event_${eventName}`, {
        data,
        timestamp: Date.now()
      });
    }
  }
  
  loadPersistentEvents() {
    // 应用启动时加载未处理的事件
    this.persistentEvents.forEach(eventName => {
      const event = wx.getStorageSync(`event_${eventName}`);
      if (event && Date.now() - event.timestamp < 60000) {
        // 1分钟内的事件重新触发
        this.emit(eventName, event.data);
      }
    });
  }
}
```

**优先级：** P3（低）  
**预计修复时间：** 1周  
**业务影响：** 低（体验优化）

#### 问题11：无缓存大小限制

**问题描述：**
本地缓存没有大小限制，长期运行可能导致存储溢出。

**改进建议：**
```javascript
// storage.js
const MAX_CACHE_SIZE = 10 * 1024 * 1024; // 10MB
const MAX_CACHE_ITEMS = 100;

class LRUCache {
  constructor(maxSize = MAX_CACHE_ITEMS) {
    this.maxSize = maxSize;
    this.cache = new Map();
  }
  
  get(key) {
    if (this.cache.has(key)) {
      const value = this.cache.get(key);
      // 移动到末尾（最近使用）
      this.cache.delete(key);
      this.cache.set(key, value);
      return value;
    }
    return null;
  }
  
  set(key, value) {
    if (this.cache.has(key)) {
      this.cache.delete(key);
    } else if (this.cache.size >= this.maxSize) {
      // 删除最久未使用的
      const firstKey = this.cache.keys().next().value;
      this.cache.delete(firstKey);
    }
    this.cache.set(key, value);
  }
}
```

**优先级：** P3（低）  
**预计修复时间：** 3-5天  
**业务影响：** 低（稳定性优化）

## 三、数据不一致案例分析

### 案例1：扫码数量不一致

**场景：**
1. 用户在小程序端扫码，数量+10
2. PC端同时查看订单，显示旧数量
3. 30秒后PC端轮询更新，数量才同步

**影响：**
- 用户可能重复扫码
- 数据统计不准确

**解决方案：**
- 扫码成功后立即通过WebSocket推送更新
- PC端实时接收并更新显示

### 案例2：任务状态冲突

**场景：**
1. 工人A在小程序端领取任务
2. 管理员同时在PC端分配该任务给工人B
3. 两端显示不同的任务负责人

**影响：**
- 任务重复执行
- 责任归属不清

**解决方案：**
- 关键操作使用乐观锁
- 冲突时后端返回错误，前端提示用户刷新

### 案例3：Token过期导致操作失败

**场景：**
1. 用户在小程序端长时间操作表单
2. Token过期，提交时返回401
3. 页面跳转到登录，表单数据丢失

**影响：**
- 用户体验差
- 数据丢失

**解决方案：**
- 实现token自动刷新
- 表单数据本地暂存

## 四、改进路线图

### 第一阶段：紧急修复（1-2周）

1. **实现请求去重机制**
   - 添加防抖/节流处理
   - 防止重复提交

2. **优化轮询策略**
   - 页面可见性检测
   - 动态调整轮询频率

3. **添加网络恢复处理**
   - 监听网络状态变化
   - 自动恢复同步

### 第二阶段：核心改进（1个月）

1. **引入WebSocket实时通信**
   - 建立WebSocket连接
   - 实现扫码、订单等关键数据实时推送

2. **实现Token自动刷新**
   - 添加refresh token机制
   - 401错误时自动刷新

3. **添加数据版本控制**
   - 缓存数据添加版本号
   - 应用更新时自动清理旧缓存

### 第三阶段：完善优化（2个月）

1. **解决多端Token冲突**
   - 后端支持多设备登录
   - 或实现设备独立token

2. **优化数据对比性能**
   - 使用版本号替代深比较
   - 增量更新机制

3. **安全加固**
   - 敏感数据加密存储
   - 传输层加密（HTTPS/WSS）

## 附录：小程序端专项分析

### A.1 小程序端发现的严重问题

#### A.1.1 Token管理缺陷

**问题描述：**
小程序端完全没有Token过期检测和自动刷新机制，完全依赖服务器返回401/403错误后才被动处理。

**代码位置：**
- `utils/storage.js` - 仅存储token，无过期时间记录
- `utils/request.js` - 401错误直接跳转登录页，无刷新尝试

**具体代码：**
```javascript
// utils/request.js 第86-92行
function handle401Error(statusCode, body, skipAuthRedirect, reject) {
  clearToken();  // 直接清除token
  if (!skipAuthRedirect) {
    triggerLoginRedirect();  // 直接跳转登录页
    reject(createError('未登录', { type: 'auth', statusCode, data: body }));
    return true;
  }
  return false;
}
```

**影响：**
- 用户操作过程中突然失效
- 表单数据可能丢失
- 用户体验极差

**与PC端对比：**
PC端同样没有Token刷新机制，但使用了Zustand状态管理，token丢失后恢复更快。

---

#### A.1.2 数据版本控制缺失

**问题描述：**
整个小程序端没有发现任何数据版本控制机制，没有ETag、Last-Modified、版本号或更新时间戳校验。

**影响：**
- 多端同时操作同一数据时产生冲突
- 本地缓存数据可能与服务器不一致
- 无法检测数据是否已过期

**风险场景：**
1. 工人A在小程序端扫码完成工序
2. 管理员在PC端修改了同一订单
3. 小程序端显示旧数据，工人B按错误状态继续操作

---

#### A.1.3 扫码同步范围有限

**问题描述：**
扫码成功后仅刷新当前页面统计，没有强制同步到其他页面和PC端。

**代码位置：**
```javascript
// pages/scan/index.js 第2176-2205行
handleScanSuccess(result) {
  // 添加到本地历史
  this.addToLocalHistory(formattedResult);
  
  // 仅刷新当前页面统计
  this.loadMyPanel(true);
  
  // 触发事件，但其他页面可能未订阅
  if (eventBus && typeof eventBus.emit === 'function') {
    eventBus.emit('SCAN_SUCCESS', result);
  }
}
```

**影响：**
- 工作台页面显示旧数据
- 订单详情页状态不更新
- PC端无法实时看到扫码结果

  }
  
  // 使用哈希比较替代JSON.stringify
  _compareData(newData, oldData) {
    return this._hashData(newData) !== this._hashData(oldData);
  }
  
  _hashData(data) {
    // 简单的哈希算法
    const str = JSON.stringify(data);
    let hash = 0;
    for (let i = 0; i < str.length; i++) {
      const char = str.charCodeAt(i);
      hash = ((hash << 5) - hash) + char;
      hash = hash & hash;
    }
    return hash;
  }
}

**6. 引入WebSocket实时同步**
```javascript
// utils/realtimeSync.js
class RealtimeSync {
  constructor() {
    this.ws = null;
    this.reconnectAttempts = 0;
    this.maxReconnectAttempts = 5;
  }
  
  connect() {
    this.ws = wx.connectSocket({
      url: 'wss://api.example.com/ws',
      header: {
        'Authorization': `Bearer ${getToken()}`
      }
    });
    
    this.ws.onOpen(() => {
      this.reconnectAttempts = 0;
      console.log('[RealtimeSync] Connected');
    });
    
    this.ws.onMessage((res) => {
      const message = JSON.parse(res.data);
      this.handleMessage(message);
    });
    
    this.ws.onClose(() => {
      this.attemptReconnect();
    });
  }
  
  handleMessage(message) {
    switch (message.type) {
      case 'ORDER_UPDATED':
        eventBus.emit(Events.ORDER_UPDATED, message.data);
        break;
      case 'SCAN_SUCCESS':
        eventBus.emit(Events.SCAN_SUCCESS, message.data);
        break;
      // ...
    }
  }
  
  attemptReconnect() {
    if (this.reconnectAttempts < this.maxReconnectAttempts) {
      this.reconnectAttempts++;
      const delay = Math.pow(2, this.reconnectAttempts) * 1000;
      setTimeout(() => this.connect(), delay);
    }
  }
}
```
