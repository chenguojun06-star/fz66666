### 1.2 发现的问题

#### 问题1：硬编码颜色值 ⚠️ 中优先级

**位置：** `pages/home/index.wxss` 等多处

**问题描述：**
多处使用了硬编码的颜色值，而不是使用 design-tokens 中定义的变量。

**代码示例：**
```css
/* ❌ 错误做法 - 硬编码 */
.reminder-badge {
  background: #ff4757;
}

/* ✅ 正确做法 - 使用CSS变量 */
.reminder-badge {
  background: var(--color-error);
}
```

**影响：**
- 主题切换困难
- 维护成本高
- 与PC端颜色不一致风险

**优化方案：**
1. 全局搜索硬编码颜色值
2. 替换为对应的CSS变量
3. 建立颜色映射表

**实施步骤：**
```bash
# 搜索硬编码颜色
 grep -r "#ff4757\|#3b82f6\|#10b981" --include="*.wxss" .

# 替换为变量
# #ff4757 → var(--color-error)
# #3b82f6 → var(--color-primary)
# #10b981 → var(--color-success)
```

#### 问题2：字体大小不统一 ⚠️ 低优先级

**位置：** `pages/home/index.wxss` 等多处

**问题描述：**
字体大小使用了多种值，没有严格遵循设计Token中的字体系统。

**设计Token定义：**
```css
--font-size-xs: 11px;  /* 辅助文字 */
```


**优化方案：**
1. 统一使用设计Token中定义的字体大小
2. 如需新增字号，先更新design-tokens.wxss

## 三、弹窗组件评估

### 3.1 弹窗实现现状

**使用方式：**
小程序主要使用 `wx.showModal` 和 `wx.showToast` 原生弹窗，以及自定义的 `modal` 组件。

**代码示例：**
```javascript
// 原生弹窗
wx.showModal({
  title: '确认删除',
  content: '确定要删除吗？',
  success: (res) => {
    if (res.confirm) {
      // 执行删除
    }
  }

// 自定义弹窗（通过App.useApp）
modal.confirm({
  title: '确认删除',
  content: '确定要删除该款式吗？',
  onOk: () => { /* 确认回调 */ }
});
```

### 3.2 发现的问题

#### 问题3：弹窗样式不统一 ⚠️ 中优先级

**问题描述：**
- 原生弹窗和自定义弹窗样式不一致
- 不同页面的确认弹窗按钮文字不统一（有的用"确定"，有的用"确认"）

**优化方案：**
1. 封装统一的弹窗工具函数
2. 统一按钮文字和样式
3. 建立弹窗使用规范

**建议封装：**
```javascript
// utils/dialog.js
const Dialog = {
  confirm({ title, content, confirmText = '确认', cancelText = '取消' }) {
    return new Promise((resolve) => {
      wx.showModal({
        title,
        content,
        confirmText,
        cancelText,
        confirmColor: '#3b82f6',
        success: (res) => resolve(res.confirm)
      });
    });
  },
  
  alert({ title, content, confirmText = '知道了' }) {
    return new Promise((resolve) => {
      wx.showModal({
        title,
        content,
        showCancel: false,
        confirmText,
        confirmColor: '#3b82f6',
        success: () => resolve()
      });
    });
  }
};
```

#### 问题4：重复方法定义 🔴 高优先级

**位置：** `pages/scan/handlers/ScanHandler.js` 第117-145行和第226-254行

**问题描述：**
`_handleSKUQuantity` 方法被定义了两次，第二次定义会覆盖第一次。

**代码示例：**
```javascript
// 第117-145行
_handleSKUQuantity(parsedData, orderDetail) {
  if (parsedData.quantity) {
    return null; // 已有数量
  }
  // ... 实现逻辑
}

// 第226-254行（重复定义，会覆盖上面的）
_handleSKUQuantity(parsedData, orderDetail) {
  if (parsedData.quantity) {
    return; // 已有数量，无需处理
  }
  // ... 不同的实现逻辑
}
```

**影响：**
- 可能导致预期外的行为
- 代码维护困难

**优化方案：**
1. 检查两个方法的差异
2. 合并为一个方法，或重命名其中一个
3. 添加单元测试验证行为

**实施步骤：**
```javascript
// 方案1：合并为一个方法
_handleSKUQuantity(parsedData, orderDetail) {
  if (parsedData.quantity) {
    return null;
  }
  // 合并两个版本的逻辑
  const batchQuantity = orderDetail.batchQuantity || orderDetail.batch_quantity || 0;
  // ... 完整逻辑
}

// 方案2：重命名第二个方法
_handleSKUQuantityV2(parsedData, orderDetail) {
  // 不同的实现
}
```

#### 问题5：空的 catch 块 ⚠️ 中优先级

**位置：** `utils/request.js` 等多处

**问题描述：**
多处使用 `null;` 或空语句作为 catch 块内容，掩盖了潜在错误。

**代码示例：**
```javascript
// ❌ 错误做法
try {
  const value = wx.getStorageSync('api_base_url');
} catch (e) {
  null;  // 静默吞掉错误
}

// ✅ 正确做法
try {
  const value = wx.getStorageSync('api_base_url');
} catch (e) {
  console.warn('Failed to get base URL:', e);
  // 或上报错误
  errorHandler.report(e, { context: 'getBaseUrl' });
}
```

**影响：**
- 调试困难
- 可能掩盖严重问题
- 无法追踪错误来源

**优化方案：**
1. 至少添加日志记录
2. 对于关键操作，上报错误
3. 给用户适当的反馈

---

#### 问题9：缺少输入验证 ⚠️ 中优先级

**位置：** `pages/scan/index.js`

**问题描述：**
部分用户输入缺少验证。

**代码示例：**
```javascript
// ❌ 缺少验证
onQuantityInput(e) {
  this.setData({ quantity: e.detail.value });
}

// ✅ 添加验证
onQuantityInput(e) {
  const value = e.detail.value;
  const num = parseInt(value, 10);
  
  if (isNaN(num) || num < 0) {
    wx.showToast({ title: '请输入有效数量', icon: 'none' });
    return;
  }
  
  if (num > 99999) {
    wx.showToast({ title: '数量不能超过99999', icon: 'none' });
    return;
  }
  
  this.setData({ quantity: num });
}
```
