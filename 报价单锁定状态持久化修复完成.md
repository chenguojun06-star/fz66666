# 报价单锁定状态持久化修复完成

> **修复日期**: 2026-01-30  
> **问题描述**: 报价单保存后锁定，但刷新页面后锁定状态丢失，又变成可编辑状态  
> **影响范围**: 后端实体类、前端报价单组件、数据库表结构

---

## 📊 问题分析

### 原始问题
用户反馈：
1. 点击"保存报单"按钮后，报价单正确锁定（按钮变灰，表单禁用）
2. 关闭样衣详情弹窗，再次打开同一个样衣
3. 报价单又变成可编辑状态（锁定状态丢失）

### 根本原因
锁定状态 `isLocked` 只保存在前端的 React 状态中，没有持久化到数据库：

```typescript
// ❌ 修复前：只在前端保存状态
const [isLocked, setIsLocked] = useState(false);

const handleSave = async () => {
  // ...保存报价单
  setIsLocked(true); // ⚠️ 只设置前端状态，刷新页面就丢失
};
```

---

## 🔧 修复方案

### 1. 数据库层：添加 `is_locked` 字段

在 `t_style_quotation` 表添加锁定状态字段：

```sql
ALTER TABLE t_style_quotation 
ADD COLUMN is_locked TINYINT DEFAULT 0 COMMENT '是否锁定(0=未锁定,1=已锁定)';
```

**字段说明**：
- `0`: 未锁定（可编辑）
- `1`: 已锁定（只读，需管理员"维护"才能解锁）

### 2. 后端实体类：添加 `isLocked` 属性

**文件**: `backend/src/main/java/com/fashion/supplychain/style/entity/StyleQuotation.java`

```java
/**
 * 是否锁定(0=未锁定，1=已锁定)
 */
private Integer isLocked;
```

### 3. 前端组件：持久化锁定状态

**文件**: `frontend/src/modules/basic/pages/StyleInfo/components/StyleQuotationTab.tsx`

#### 3.1 加载时读取锁定状态

```typescript
const fetchData = async () => {
  // ...获取报价单数据
  if (existing && 'isLocked' in existing) {
    setIsLocked(existing.isLocked === 1); // ✅ 从后端读取锁定状态
  }
};
```

#### 3.2 保存时写入锁定状态

```typescript
const handleSave = async () => {
  const data = {
    ...(quotation || {}),
    ...values,
    styleId,
    isLocked: 1, // ✅ 保存到后端
  };
  await api.post('/style/quotation', data);
  setIsLocked(true);
  message.success('保存成功，报价单已锁定');
};
```

#### 3.3 解锁时更新后端

```typescript
const handleUnlock = async () => {
  const payload = {
    ...quotation,
    isLocked: 0, // ✅ 解锁状态写入后端
  };
  await api.put('/style/quotation', payload);
  setIsLocked(false);
  message.success('已解锁，可以编辑');
};
```

## 📐 数据流转

### 保存流程
```
用户点击"保存报单"
    ↓
前端: handleSave() 
    ↓
发送 POST /style/quotation { isLocked: 1 }
    ↓
后端: StyleQuotationService.save()
    ↓
数据库: UPDATE t_style_quotation SET is_locked = 1
    ↓
前端: setIsLocked(true) → UI变灰色
```

### 加载流程
```
用户打开样衣详情
    ↓
前端: fetchData()
    ↓
发送 GET /style/quotation?styleId=xxx
    ↓
后端: StyleQuotationService.getByStyleId()
    ↓
数据库: SELECT * FROM t_style_quotation WHERE style_id = xxx
    ↓
后端返回: { ..., isLocked: 1 }
    ↓
前端: setIsLocked(res.data.isLocked === 1) → UI保持灰色
```

### 解锁流程
```
管理员点击"维护"按钮
    ↓
前端: handleUnlock()
    ↓
发送 PUT /style/quotation { isLocked: 0 }
    ↓
后端: StyleQuotationService.update()
    ↓
数据库: UPDATE t_style_quotation SET is_locked = 0
    ↓
前端: setIsLocked(false) → UI恢复可编辑
```

## 📊 技术细节

### 数据库设计
- **字段类型**: `TINYINT`（占用1字节，节省空间）
- **默认值**: `0`（新建报价单默认未锁定）
- **可空**: `YES`（兼容旧数据）

### 前后端一致性
| 层级 | 字段名 | 类型 | 值含义 |
|------|--------|------|--------|
| 数据库 | `is_locked` | TINYINT | 0=未锁定, 1=已锁定 |
| 后端实体 | `isLocked` | Integer | 0=未锁定, 1=已锁定 |
| 前端状态 | `isLocked` | boolean | false=未锁定, true=已锁定 |

### 类型转换
```typescript
// 后端 → 前端
setIsLocked(res.data.isLocked === 1); // Integer → boolean

// 前端 → 后端
const data = { isLocked: 1 };        // boolean → Integer
const data = { isLocked: 0 };
```

## 🔒 安全性保障

### 权限控制
- **保存报单**: 任何有权限的用户都可以保存并锁定
- **维护（解锁）**: 需要管理员权限（通过"维护"按钮）

### 数据完整性
- 锁定状态与报价单数据强绑定（同一张表）
- 通过数据库事务保证一致性
- 解锁需显式操作，不会意外丢失

### 防误操作
- 保存后自动锁定 → 防止意外修改已审核的报价单
- 解锁需确认操作 → 防止误点
- 锁定状态持久化 → 防止刷新页面后误以为可编辑

## 📌 注意事项

1. **⚠️ 旧数据兼容性**: 已有的报价单 `is_locked` 默认为 `0`（未锁定），需要重新保存才会锁定

2. **⚠️ 前端状态同步**: 解锁后需要刷新数据（调用 `fetchData()`），确保 UI 与后端状态一致

3. **⚠️ 并发控制**: 多用户同时编辑同一个报价单时，可能产生冲突。建议添加乐观锁（version 字段）

4. **⚠️ 数据库迁移**: 生产环境部署时需执行 SQL 脚本添加字段
