### 2. PatternRevisionController（10 → 7端点，-3）

**问题**：纸样修改工作流有4个独立端点（submit/approve/reject/complete），都是状态转换操作。

**解决方案**：统一工作流端点 `POST /{id}/workflow?action=xxx`

#### 新端点设计

```java
POST /api/pattern-revision/{id}/workflow
Query参数：
  - action: 操作类型（submit/approve/reject/complete）
Body参数（可选）：
  - comment: 审核意见（用于approve和reject）
```

#### 智能路由逻辑

```java
@PostMapping("/{id}/workflow")
public Result<?> workflow(
        @PathVariable String id,
        @RequestParam String action,
        @RequestBody(required = false) Map<String, String> params) {
    
    switch (action.toLowerCase()) {
        case "submit":
            boolean submitSuccess = patternRevisionService.submitForApproval(id);
            return submitSuccess ? Result.success() : Result.fail("提交失败");
            
        case "approve":
            String approveComment = params != null ? params.getOrDefault("comment", "") : "";
            boolean approveSuccess = patternRevisionService.approve(id, approveComment);
            return approveSuccess ? Result.success() : Result.fail("审核失败");
            
        case "reject":
            String rejectComment = params != null ? params.getOrDefault("comment", "") : "";
            if (!StringUtils.hasText(rejectComment)) {
                return Result.fail("请填写拒绝原因");
            }
            boolean rejectSuccess = patternRevisionService.reject(id, rejectComment);
            return rejectSuccess ? Result.success() : Result.fail("拒绝失败");
            
        case "complete":
            boolean completeSuccess = patternRevisionService.complete(id);
            return completeSuccess ? Result.success() : Result.fail("完成失败");
            
        default:
            return Result.fail("不支持的操作: " + action);
    }
}
```

#### 权限控制（多权限合并）

```java
@PreAuthorize("hasAnyAuthority('PATTERN_REVISION_SUBMIT', 'PATTERN_REVISION_APPROVE', 'PATTERN_REVISION_REJECT', 'PATTERN_REVISION_COMPLETE')")
```

```java

    return workflow(id, "submit", null);
}

// 类似的废弃端点：

```

**工作流状态转换图**：

### 3. ProductWarehousingController（9 → 8端点，-1）

**问题**：报修统计有2个端点（`/repair-stats` GET和 `/repair-stats/batch` POST），功能相同只是单个vs批量。

**解决方案**：统一端点同时支持GET和POST

#### 新端点设计

```java
// 同时支持GET和POST
@RequestMapping(value = "/repair-stats", method = {RequestMethod.GET, RequestMethod.POST})
```

#### 智能路由逻辑

```java
@RequestMapping(value = "/repair-stats", method = {RequestMethod.GET, RequestMethod.POST})
public Result<?> repairStats(
        @RequestParam(required = false) Map<String, Object> params,
        @RequestBody(required = false) Map<String, Object> body) {
    
    // 智能路由：根据请求方式和参数选择单个或批量处理
    if (body != null && !body.isEmpty()) {
        // POST请求 + body存在 = 批量处理
        return Result.success(productWarehousingOrchestrator.batchRepairStats(body));
    } else {
        // GET请求或POST无body = 单个处理
        return Result.success(productWarehousingOrchestrator.repairStats(params != null ? params : body));
    }
}
```

### 4. MaterialReconciliationController（8 → 6端点，-2）

**问题**：状态操作有2个独立端点（`/update-status` 更新状态，`/return` 退回上一状态）。

**解决方案**：统一状态操作端点 `POST /{id}/status-action?action=xxx`

#### 新端点设计

```java
POST /api/finance/material-reconciliation/{id}/status-action
Query参数：
  - action: 操作类型（update/return）
  - status: 目标状态（用于update，如：APPROVED/REJECTED）
  - reason: 退回原因（用于return，可选）
```

#### 智能路由逻辑

```java
@PostMapping("/{id}/status-action")
public Result<?> statusAction(
        @PathVariable String id,
        @RequestParam String action,
        @RequestParam(required = false) String status,
        @RequestParam(required = false) String reason) {
    
    switch (action.toLowerCase()) {
        case "update":
            if (status == null || status.trim().isEmpty()) {
                return Result.fail("status不能为空");
            }
            String updateMessage = reconciliationStatusOrchestrator.updateMaterialStatus(id, status);
            return Result.successMessage(updateMessage);
            
        case "return":
            String returnMessage = reconciliationStatusOrchestrator.returnMaterialToPrevious(id, reason);
            return Result.successMessage(returnMessage);
            
        default:
            return Result.fail("不支持的操作: " + action);
    }
}
```

#### 使用示例

```typescript
// 更新状态
POST /api/finance/material-reconciliation/123/status-action?action=update&status=APPROVED

// 退回上一状态
POST /api/finance/material-reconciliation/123/status-action?action=return&reason=信息不完整
```

## 🚀 技术实现亮点

### 1. 同步/异步统一模式（StyleBom）

**核心思想**：用布尔参数区分执行方式，而非路径区分

```java
// 旧方式：路径区分
POST /{styleId}/sync-material-database       → 同步执行
POST /{styleId}/sync-material-database/async → 异步执行

// 新方式：参数区分
POST /{styleId}/sync-material-database?async=false → 同步执行
POST /{styleId}/sync-material-database?async=true  → 异步执行
```

**优势**：
- RESTful原则：同一资源同一路径
- 参数语义化：async 比 /async 更清晰
- 易扩展：可添加更多执行选项（priority、timeout等）

### 2. 工作流状态机模式（PatternRevision）

**核心思想**：用action参数表示状态转换动作

```
状态转换 = f(当前状态, action)

DRAFT + submit   → PENDING
PENDING + approve → APPROVED
PENDING + reject  → REJECTED
APPROVED + complete → COMPLETED
```

**实现方式**：
```java
// 所有状态转换集中在Service层
switch (action) {
    case "submit":   → submitForApproval(id)
    case "approve":  → approve(id, comment)
    case "reject":   → reject(id, comment)
    case "complete": → complete(id)
}
```

**优势**：
- 状态机可视化：清晰的状态转换图
- 权限细粒度：hasAnyAuthority 支持多权限
- 易维护：状态逻辑集中在Service层

### 3. GET/POST双模式（ProductWarehousing）

**核心思想**：同一端点支持多种HTTP方法，根据请求内容智能路由

```java
@RequestMapping(value = "/repair-stats", method = {RequestMethod.GET, RequestMethod.POST})
public Result<?> repairStats(
        @RequestParam(required = false) Map<String, Object> params,  // GET方式
        @RequestBody(required = false) Map<String, Object> body) {    // POST方式
    
    if (body != null && !body.isEmpty()) {
        // POST + body → 批量处理
        return batchRepairStats(body);
    } else {
        // GET 或 POST无body → 单个处理
        return singleRepairStats(params);
    }
}
```

**优势**：
- RESTful友好：同一资源同一路径
- 自适应：根据请求自动选择处理方式
- 向后兼容：完全兼容旧版GET请求

### 4. 参数化状态操作（MaterialReconciliation）

**核心思想**：用Query参数传递操作类型和参数，简化Body结构

```java
// 旧方式：Body传参
POST /update-status
Body: { id: "123", status: "APPROVED" }

POST /return
Body: { id: "123", reason: "信息不完整" }

// 新方式：Query参数传参
POST /123/status-action?action=update&status=APPROVED
POST /123/status-action?action=return&reason=信息不完整
```

**优势**：
- 参数可见：Query参数在URL中直接可见
- 易调试：浏览器/日志可直接看到参数
- 易测试：Query参数比Body更容易构造
