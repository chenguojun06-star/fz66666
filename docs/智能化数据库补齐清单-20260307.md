## 2. 租户真实痛点对应的智能化方向

### 2.1 老板 / 租户主账号的痛点

- 不知道订单风险到底为什么发生
- 不知道该先保交期、保利润还是保回款
- 不知道哪家工厂适合什么款
- 系统能提示风险，但不能证明“这个建议以前有用”

### 2.2 跟单 / 生产管理的痛点

- 风险太多，不知道先处理哪个
- 工厂老说能做，结果做不稳
- 同类异常反复出现，但系统没有固化处理手法

### 2.3 工厂 / 车间的痛点

- 工人只看名字，不知道适配工序
- 排单靠经验，容易把复杂工序压给不稳定的人/厂
- 返工和卡点没有形成训练数据

## 4. P0 必补表设计

### 4.1 `t_intelligence_feedback_reason`

### 为什么要补

现有 [backend/src/main/java/com/fashion/supplychain/intelligence/orchestration/FeedbackLearningOrchestrator.java](backend/src/main/java/com/fashion/supplychain/intelligence/orchestration/FeedbackLearningOrchestrator.java) 只知道“接没接受建议”，不知道“为什么”。

没有这个表，系统学不到：

- 这个租户不接受排产建议，是因为工厂不受控
- 这个租户不关心扫码异常，而更关心回款节奏
- 某类建议经常被拒绝，是因为建议不落地而不是算法不准

### 建议字段

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT | 主键 |
| tenant_id | BIGINT | 租户ID |
| prediction_id | VARCHAR(64) | 关联预测/建议ID |
| suggestion_type | VARCHAR(64) | 建议类型：DELIVERY/PRECHECK/SCHEDULING/INOUT/FINANCE |
| accepted | TINYINT(1) | 是否接受 |
| reason_code | VARCHAR(64) | 原因编码 |
| reason_text | VARCHAR(500) | 拒绝或接受原因文本 |
| order_id | BIGINT | 订单ID，可空 |
| stage_name | VARCHAR(64) | 阶段名，可空 |
| process_name | VARCHAR(64) | 工序名，可空 |
| operator_id | VARCHAR(64) | 操作人 |
| operator_name | VARCHAR(64) | 操作人名 |
| create_time | DATETIME | 创建时间 |
| update_time | DATETIME | 更新时间 |
| delete_flag | INT | 逻辑删除 |

### 建议原因码

- `FACTORY_UNCONTROLLABLE` 工厂不可控
- `PRIORITY_NOT_MATCH` 当前优先级不匹配
- `CAPACITY_NOT_ENOUGH` 产能不足
- `CASHFLOW_FIRST` 当前要先保回款
- `MARGIN_FIRST` 当前要先保利润
- `HUMAN_OVERRIDE` 人工经验覆盖

### 编排挂接点

- 写入：`FeedbackLearningOrchestrator`
- 聚合分析：新增 `FeedbackReasonViewOrchestrator`

### 4.2 `t_intelligence_pain_point`

### 为什么要补

现在风险和原因更多是即时计算，没有把“租户真正反复痛的点”沉淀出来。

缺这个表会导致：

- 系统每次都像第一次发现问题
- 没法排序“高频痛点”
- 没法做租户经营复盘

### 建议字段

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT | 主键 |
| tenant_id | BIGINT | 租户ID |
| pain_code | VARCHAR(64) | 痛点编码 |
| pain_name | VARCHAR(128) | 痛点名称 |
| pain_level | VARCHAR(16) | HIGH/MEDIUM/LOW |
| business_domain | VARCHAR(32) | PRODUCTION/FINANCE/WAREHOUSE/FACTORY |
| trigger_count | INT | 触发次数 |
| affected_order_count | INT | 影响订单数 |
| affected_amount | DECIMAL(12,2) | 影响金额，可空 |
| latest_trigger_time | DATETIME | 最近触发时间 |
| root_reason_summary | VARCHAR(500) | 根因摘要 |
| current_status | VARCHAR(16) | OPEN/IMPROVING/CLOSED |
| create_time | DATETIME | 创建时间 |
| update_time | DATETIME | 更新时间 |
| delete_flag | INT | 逻辑删除 |

### 推荐 pain_code

- `DELAY_REPEAT` 同类拖期反复出现
- `FACTORY_UNSTABLE` 工厂执行不稳定
- `LOW_MARGIN_REPEAT` 低毛利订单持续增加
- `SCAN_EXCEPTION_REPEAT` 扫码异常重复发生
- `CASHFLOW_PRESSURE` 回款压力偏高
- `MATERIAL_SHORTAGE_REPEAT` 物料短缺反复发生

### 编排挂接点

- 生成/刷新：新增 `PainPointAggregationOrchestrator`
- 消费：`TenantIntelligenceProfileViewOrchestrator`、`DailyBriefOrchestrator`

## 8. 对应编排拆分原则

新增数据库后，必须继续遵守独立编排铁律：

- 配置类：单独 system 编排
- 学习类：单独 intelligence 编排
- 效果回流类：单独 intelligence 编排
- 视图聚合类：单独 intelligence 编排

禁止做法：

- 在一个编排器里同时做“配置写入 + 学习计算 + 页面聚合”
- 在已有大编排器里继续硬塞新表逻辑
- 前端一个 service 混多个独立领域接口

---

## 9. 数据库发布注意事项

仓库明确约束：云端 `FLYWAY_ENABLED=false`。

所以所有新增智能化表都必须同时满足两步：

1. 本地写 `V*.sql` 迁移脚本纳入版本控制
2. 云端在微信云托管控制台手工执行 SQL，并留存执行记录

如果只写本地迁移，不做云端 SQL，智能化功能会出现“本地可用、线上失效”的假成功。

---

## 10. 最终建议

如果只允许做 3 件最有价值的事，优先顺序如下：

1. 先补 `t_intelligence_feedback_reason`
原因：没有“为什么接受/拒绝”，租户个性化永远学不深。

2. 再补 `t_intelligence_solution_playbook + t_intelligence_solution_effect`
原因：没有“方案”和“效果”，智能化只是提醒器，不是解决方案系统。

3. 再补 `t_factory_skill_matrix`
原因：这是最能直接改善派单、交期、返工和毛利的智能资产表。

结论：

当前最该补的，不是更多智能页面，而是“建议原因、解决方案、效果回流、工厂技能矩阵”这四类数据库资产。
