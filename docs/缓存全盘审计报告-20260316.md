# 缓存全盘审计报告

审计时间：2026-03-16  
审计范围：backend、frontend 现有缓存实现与主要热点调用链  
审计目标：识别缓存失效缺口、跨租户缓存污染风险、无界内存驻留风险，并先修复会直接影响正确性的缓存问题。

## 一、审计结论

本轮已完成对仓库内主要缓存实现的全盘梳理，并优先修复了 4 处确定会影响数据正确性或多租户隔离的缓存问题。

当前系统缓存规模粗统计如下：

- 后端 `@Cacheable`：12 处
- 后端 Caffeine 本地缓存：3 处
- 后端手写 Redis 读写/删除：36 处
- 前端 `localStorage`：89 处
- 前端 `sessionStorage`：12 处
- 前端模块级 `Map`/内存缓存：64 处

整体判断：

1. 前端大量缓存属于界面偏好、分页状态、视图模式与短期请求去重，数量多但大多不是数据一致性高风险点。
2. 后端缓存点虽然数量不算夸张，但更接近真实业务数据，错误会直接表现为跨租户串数据、写后读旧数据或统计口径异常，因此优先级更高。
3. 这次修完后，当前最明确的 correctness 风险已经收敛，剩余主要是“无 TTL 的内存缓存”和“前端模块级缓存的生命周期治理”两类结构性问题。

## 二、缓存分类盘点

### 1. 后端 Spring Cache / Redis

核心位置：`backend/src/main/java/com/fashion/supplychain/config/RedisConfig.java`

当前已确认的缓存区：

- `user`：2 小时
- `dict`：1 小时
- `style`：10 分钟
- `order`：5 分钟
- `permission`：1 小时
- `daily-brief`：5 分钟
- `dataCenter`：默认 TTL 30 分钟

当前机制：Redis 不可用时会自动降级为 cache miss，不因为缓存故障直接中断主链路。

### 2. 后端 Caffeine 本地缓存

核心位置：`backend/src/main/java/com/fashion/supplychain/template/service/impl/TemplateLibraryServiceImpl.java`

已确认缓存：

- `progressTemplateCache`
- `processPriceTemplateCache`

当前策略：10 分钟过期，最大 200 条。

### 3. 后端手写 Redis / 内存缓存

代表位置：

- `backend/src/main/java/com/fashion/supplychain/style/service/impl/StyleBomServiceImpl.java`
- `backend/src/main/java/com/fashion/supplychain/config/SecurityConfig.java`
- `backend/src/main/java/com/fashion/supplychain/intelligence/service/AiAdvisorService.java`

已确认用途：

- BOM 列表缓存
- 登录后租户信息补全缓存
- AI 每租户每日配额计数

### 4. 前端持久化缓存

主要形态：

- `localStorage`：登录 token、用户信息、主题、列表视图模式、列配置、搜索历史、消息忽略状态
- `sessionStorage`：错误边界限流、页面刷新保护、采购页查询条件暂存

当前判断：大部分属于 UX / 状态保留缓存，不是本轮优先修复对象。

### 5. 前端模块级内存缓存

代表位置：

- `frontend/src/utils/api/production.ts`
- `frontend/src/smart/components/WorkerPerformanceBadge.tsx`
- `frontend/src/stores/productionBoardStore.ts`

主要用途：

- 订单冻结状态短 TTL 去重
- 员工画像重复请求合并
- 生产进度球全局共享缓存

当前判断：这些缓存主要服务性能和交互稳定性，短期内可接受，但后续需要继续统一失效边界。

## 三、本轮已修复问题

### 1. BOM 写后缓存未正确失效

文件：`backend/src/main/java/com/fashion/supplychain/style/service/impl/StyleBomServiceImpl.java`

问题：

- BOM 实际缓存键格式为 `style:bom:<styleId>:<variant>:<variant>`。
- 旧的 `clearBomCache(styleId)` 只删除 `style:bom:<styleId>` 单键。
- 结果是 BOM 更新后，真实列表缓存可能仍然命中旧值。

修复：

- 将删除逻辑改为 `deleteByPattern("style:bom:" + styleId + ":*")`。

收益：

- BOM 写后读旧数据问题被直接消除。
- 不再依赖调用方恰好命中某个不存在的基础 key。

### 2. 模板库本地缓存缺少租户隔离

文件：`backend/src/main/java/com/fashion/supplychain/template/service/impl/TemplateLibraryServiceImpl.java`

问题：

- `progressTemplateCache` 与 `processPriceTemplateCache` 原先只按 `styleNo` 缓存。
- 多租户下如果不同租户存在相同款号，可能命中到别的租户模板。

修复：

- 新增 `buildTemplateCacheKey(styleNo)`。
- 缓存键改为 `tenantPart:stylePart`。
- 失效逻辑同步改为按租户维度失效。

收益：

- 消除同款号跨租户模板串用风险。
- 保证模板读取与失效语义一致。

### 3. DataCenter 查询缓存缺少租户前缀

文件：`backend/src/main/java/com/fashion/supplychain/datacenter/service/impl/DataCenterQueryServiceImpl.java`

问题：

- `enabledStylesCount`、`materialPurchasesCount`、`productionOrdersCount`、`style/bom/size/attachments` 等缓存键原本未统一带租户维度。
- 多租户部署下，数据中心统计和款式关联缓存存在串租户风险。

修复：

- 所有 `@Cacheable` key 统一改为 `tenantId + ':' + ...`。

收益：

- 数据中心聚合统计不再跨租户复用。
- 款式/BOM/尺码/附件缓存语义与多租户模型对齐。

### 4. AI 日报建议缓存缺少租户隔离

文件：`backend/src/main/java/com/fashion/supplychain/intelligence/service/AiAdvisorService.java`

问题：

- `getDailyAdvice(contextSummary)` 原先只按 `contextSummary` 做缓存。
- 两个租户若摘要文本偶然相同，可能共用同一份 AI 建议结果。

修复：

- 缓存键改为 `tenantId + ':' + contextSummary`。

收益：

- AI 日报结果与租户数据边界保持一致。
- 避免跨租户命中“表面正常、实则串数据”的隐蔽问题。

## 四、已识别但暂未修改的风险

### 1. `tenantInfoCache` 为无 TTL 常驻内存缓存

文件：`backend/src/main/java/com/fashion/supplychain/config/SecurityConfig.java`

现状：

- `tenantInfoCache` 使用 `ConcurrentHashMap<String, String>` 存 userId 对应的租户信息。
- 当前无 TTL、无最大容量限制。
- 虽然已有少量手动失效逻辑，但整体仍偏向“永久驻留直到进程重启”。

为什么这次没直接改：

- 它位于认证与租户补全主链路，调整不当会影响全站登录态解析。
- 当前更紧急的是修 correctness 问题，而不是立即替换认证链路中的本地缓存实现。

建议后续方案：

- 改为 Caffeine，增加 `expireAfterWrite` 与 `maximumSize`。
- 或迁移到 Redis，统一生命周期与显式失效策略。

### 2. AI 配额计数 `dailyCounters` 仍是进程内缓存

文件：`backend/src/main/java/com/fashion/supplychain/intelligence/service/AiAdvisorService.java`

现状：

- 每租户每日配额计数存于 `ConcurrentHashMap<String, AtomicInteger>`。
- 单实例下问题不大，多实例部署下不具备全局一致性。

当前判断：

- 它更像“轻量限流”而非财务级强一致配额，不是本轮阻塞项。
- 如果后续 AI 计费或额度治理变严，应迁到 Redis 原子计数。

### 3. 前端模块级 `Map` 缓存未统一租户维度

代表位置：

- `frontend/src/utils/api/production.ts`
- `frontend/src/smart/components/WorkerPerformanceBadge.tsx`

现状：

- 这类缓存通常按订单 ID、员工名等业务键缓存，不显式拼租户维度。
- 在浏览器单租户常规操作下问题较小，但如果同一浏览器频繁切租户、且业务 ID 重叠，理论上仍有短期串值窗口。

当前判断：

- 因为这些缓存多数 TTL 很短，或随页面刷新自然失效，暂不作为第一优先级。
- 若后续要继续做缓存治理，优先统一“租户切换时清空模块级缓存”的前端基建。

## 五、对系统的实际帮助

本轮缓存治理的直接价值有三点：

1. 修掉了一个确定存在的写后缓存失效 bug，避免 BOM 数据保存后仍读到旧值。
2. 修掉了三条跨租户缓存污染链路，降低多租户系统最隐蔽的一类数据串读风险。
3. 把仓库里的缓存实现按类型归类，后续再做二轮治理时，不需要从头重新摸排。

## 六、建议的后续动作

建议下一轮缓存治理按以下顺序推进：

1. 将 `SecurityConfig.tenantInfoCache` 改为带 TTL 和容量上限的 Caffeine 缓存。
2. 评估 `AiAdvisorService.dailyCounters` 是否需要迁移到 Redis 原子计数。
3. 增加“前端切租户时统一清空模块级内存缓存”的基础设施。
4. 对关键业务查询继续保持“缓存键必须显式带租户维度”的审查规则。

## 七、附注

本报告基于当前仓库静态审计与本轮修复结果整理。

其中前端 `localStorage`、`sessionStorage`、`Map` 的数量统计包含大量 UI 状态与请求去重用途，不等同于同等级风险点；真正优先级仍以是否影响业务正确性、多租户隔离和写后读一致性为准。
