# 全系统双端智能中枢蓝图（2026-03-07）

> 目标不是“做几个智能面板”，而是把系统升级成一个持续感知、持续判断、持续行动、持续学习、持续适配租户的智能中枢。
>
> 范围覆盖：PC 端 + 小程序端 + 后端全链路，业务域覆盖开发、生产、出入库、财务结算、系统治理。
>
> 原则：不破坏既有 Controller → Orchestrator → Service → Mapper 四层架构，不把智能逻辑散落到 Controller 和 Service 里。

---

## 1. 最终目标定义

系统要从“静态业务系统”升级为“活系统”，必须具备 5 个能力：

1. 会感知
- 所有关键业务变化都能被系统及时捕捉，而不是只写库不留痕。

2. 会判断
- 能把订单、工厂、工人、款式、库存、财务等多信号融合成态势判断，而不是单一规则打标签。

3. 会行动
- 不停留在预警和报表，要能生成待办、通知、推荐动作、自动修复动作。

4. 会记住
- 把成功经验、失败案例、返工原因、质检缺陷、交付偏差沉淀成租户自己的智能记忆。

5. 会进化
- 每次建议和动作都有反馈闭环，系统根据采纳率、误报率、修复效果逐步优化策略。

## 3. 全系统智能中枢目标架构

## 3.1 架构原则

- 所有智能能力仍然通过新增或扩展 Orchestrator 承载。
- 所有写操作仍由 Orchestrator 统一加事务。
- 所有智能功能按租户级 Feature Flag 灰度开启。
- 所有建议必须“可解释、可追溯、可关闭、可回滚”。
- 所有自动动作必须分级：只提示、建议执行、软拦截、自动执行。

## 3.2 中枢分层

### A. 感知层 Awareness Layer

职责：从全业务系统实时采集信号。

事件来源：
- 开发端：款式创建、BOM 缺失、工序模板变更、报价异常
- 生产端：扫码成功、扫码失败、跳序、停滞、卡点、返工
- 出入库：库存不足、异常入库、出库延迟、物料短缺
- 财务：差异结算、重复结算风险、负利润订单、异常审批
- 系统：角色缺权限、词典脏数据、租户配置缺失、应用开通异常
- 小程序：离线缓存积压、扫码模式切换、失败重试、待同步数据

建议新增编排器：
- IntelligenceSignalOrchestrator

职责边界：
- 不直接做业务处理，只负责统一产出智能事件。

### B. 态势层 Situation Layer

职责：把离散事件融合为业务态势。

态势对象：
- 订单态势
- 工厂态势
- 工人态势
- 款式态势
- 财务态势
- 租户态势

建议新增编排器：
- OrderSituationOrchestrator
- FactorySituationOrchestrator
- WorkerSituationOrchestrator
- FinanceSituationOrchestrator
- TenantSituationOrchestrator

融合输入：
- 已有 DeliveryRisk
- 已有 SmartPrecheck
- 已有 LivePulse
- 已有 SmartNotification
- 已有 SelfHealing
- 已有 WorkerProfile
- 已有 AnomalyDetection

输出：
- riskLevel
- healthScore
- urgentActions
- nextBestActions
- explainReasons

### C. 行动层 Action Layer

职责：把态势转成下一步动作，而不是只停在显示上。

动作类型：
- 待办任务
- 主动通知
- 页面引导
- 推荐派工
- 推荐排期
- 推荐补料
- 推荐复检
- 自动修复
- 自动生成采购建议
- 自动生成财务复核任务

建议新增编排器：
- ActionCenterOrchestrator
- FollowupTaskOrchestrator
- SmartEscalationOrchestrator

动作分级：
- L1：仅提示
- L2：建议执行
- L3：软拦截，可手动继续
- L4：自动执行并写审计日志

### D. 记忆层 Memory Layer

职责：让系统有“租户自己的经验库”。

记忆内容：
- 历史延期案例
- 返工与缺陷案例
- 工艺最佳实践
- 类似款式报价经验
- 供应商表现记录
- 工厂/工人协作效果
- 财务异常复核经验

建议新增编排器：
- IntelligenceMemoryOrchestrator
- CaseRecallOrchestrator
- KnowledgeRecallOrchestrator

### E. 学习层 Learning Layer

职责：根据结果回写，不断提升建议质量。

学习输入：
- 建议是否被采纳
- 采纳后是否改善
- 是否误报
- 处理耗时
- 偏差大小
- 用户手工改判原因

建议新增编排器：
- LearningLoopOrchestrator
- SuggestionEffectOrchestrator
- StrategyTuningOrchestrator

## 3.3 开源增强栈推荐（让系统更像“活的大脑”）

目标不是把一堆 AI 项目堆进系统，而是给现有智能中枢补上 5 类真正缺的能力：

1. 模型总线
2. 上下文记忆
3. 可靠动作执行
4. 可观测与评估
5. 内部协作入口

### A. 第一梯队：建议优先接入

#### 1. LiteLLM

定位：模型网关层。

适合接入位置：
- 放在 Ollama、外部兼容 OpenAI 模型、后续备用模型供应商之前
- 作为 IntelligenceBrainOrchestrator、NlQueryOrchestrator、后续 ActionCenter 的统一模型出口

核心价值：
- 统一 OpenAI 格式调用 100+ 模型与推理服务
- 支持路由、重试、降级、成本跟踪、虚拟密钥、网关治理
- 便于后续做“同一请求自动切换模型”而不改业务 Orchestrator

为什么适合本系统：
- 你现在的智能能力已经很多，但模型调用未来一定会越来越分散
- LiteLLM 可以把“大脑的神经总线”统一起来，避免 Java 业务层直接绑死某一个模型

使用建议：
- 作为统一模型网关，不要把业务逻辑写进 LiteLLM
- 业务判断仍放在现有 Java Orchestrator 中

#### 2. Haystack

定位：上下文工程与 Agent/RAG 编排层。

适合接入位置：
- 作为独立 Python sidecar 服务
- 负责复杂检索、工具路由、记忆召回、长上下文拼装
- 通过 HTTP 或 MCP 给现有 Spring Boot intelligence 模块调用

核心价值：
- 对检索、路由、记忆、生成链路控制非常透明
- 比“纯聊天代理框架”更适合生产级、可解释、可调试的上下文编排
- 可把款式、工序、缺陷、结算、供应商、历史案例拼成高质量上下文

为什么适合本系统：
- 你们系统是强业务结构化系统，不缺 CRUD，缺的是“把多个业务域记忆拼成上下文”的能力
- Haystack 很适合做订单态势、根因分析、相似案例召回、报价解释

使用建议：
- 不要拿它替代主业务后端
- 只承担上下文工程、召回、工具编排、解释生成

#### 3. Temporal

定位：可靠动作执行神经系统。

适合接入位置：
- 接在未来 ActionCenterOrchestrator 后面
- 用于跨小时、跨天、带重试、带补偿、带人工确认的长流程动作

最适合的业务动作：
- 高风险订单升级跟进
- 停滞订单 24 小时后再次催办
- 物料短缺自动生成补料待办并等待处理结果
- 财务差异复核任务派发与超时升级
- 小程序现场异常处理回执追踪

为什么适合本系统：
- 真正“活起来”的系统，不是会看风险，而是会持续跟进直到事情被解决
- 你们现有 Spring 事务适合短事务，不适合长期动作编排
- Temporal 正好补足这根神经

使用建议：
- 把它当“动作耐久层”，不是业务主引擎
- 关键规则和权限仍在 Java Orchestrator 内判定

#### 4. OpenLIT 或 Langfuse（二选一为主，另一个可补）

定位：AI 可观测、成本、评估、提示词治理层。

推荐原则：
- 偏 LLM 调用与评测闭环：优先 Langfuse
- 偏 OpenTelemetry 原生、全链路接入、指标统一：优先 OpenLIT

OpenLIT 的优势：
- OpenTelemetry 原生
- 支持模型、向量库、框架、GPU 监控
- 自带评估、提示词管理、密钥管理、守护栏能力

为什么适合本系统：
- 你要的是“越来越聪明”，那就必须知道哪类建议被采纳、哪类建议经常误报、哪类模型成本过高
- 没有这层，系统只会越来越复杂，不会越来越聪明

使用建议：
- 不与业务数据库混写
- 单独作为 AI 运行观测面板

### B. 第二梯队：适合作为增强层，不建议先做核心依赖

#### 5. RAGFlow

定位：企业知识引擎与重文档 RAG 平台。

适合场景：
- 各类操作手册、工艺文档、质检标准、财务制度、培训文档、供应商规范的统一 ingest
- 需要 OCR、复杂 PDF、扫描件、异构文档解析时非常有价值

为什么值得接：
- 你们仓库文档很多，未来租户自己的制度、质检标准、报价规则也会越来越多
- RAGFlow 对复杂文档理解、切块、引用溯源比自己手搓更省时间

为什么不建议做第一核心：
- 它更适合知识层，不适合做业务主控制器
- 资源占用相对更重，接入复杂度高于 LiteLLM 和 OpenLIT

建议接法：
- 做“知识中枢”而不是“业务中枢”
- 先接开发规范、质检 SOP、工艺模板、财务制度四类文档

#### 6. Open WebUI

定位：内部 AI 协作入口与运营使用界面。

适合场景：
- 老板、跟单、厂长、财务主管、客服的统一 AI 工作台
- 内部问答、跨角色智能问询、模型对比、文档助手、知识对话

为什么有用：
- 可以快速形成“系统外壳”，让不同角色都能直接和 AI 交互
- 对试验租户、超级管理员、培训场景很友好

为什么不建议做核心底座：
- 它更偏入口 UI，不是你的业务大脑本体
- 当前许可包含品牌保留要求，适合内部用，不适合直接作为你们业务主产品外壳

建议接法：
- 作为内部运维和运营助手门户
- 不直接嵌入主业务系统核心路径

### C. 第三梯队：当前阶段不建议作为主干

不建议把下列类型作为核心主脑：

- 纯低代码 Agent 编排平台
- 纯聊天工作流平台
- 只会搭积木但缺少可靠事务和权限体系的自动化平台

原因很直接：
- 你们是强业务、强事务、强租户隔离系统
- 真正的主脑必须服从现有 Orchestrator 架构、权限模型、审计要求
- 这些平台适合做试验，不适合接管生产、库存、结算主流程

## 3.4 推荐落地组合

### 最推荐的开源组合

- Ollama：本地/私有模型运行
- LiteLLM：模型网关与路由总线
- Qdrant：向量记忆与相似案例召回
- Haystack：上下文编排与工具链路
- OpenLIT 或 Langfuse：可观测与评估闭环
- Temporal：动作执行与跟进闭环
- RAGFlow：知识文档中枢

### 这套组合分别对应“大脑”的哪一部分

- Ollama：大脑算力
- LiteLLM：神经总线
- Qdrant：长期记忆
- Haystack：思考与检索编排
- Temporal：动作神经与持续执行
- OpenLIT / Langfuse：自我观察与纠偏
- RAGFlow：文档知识皮层

### 对你们系统最直接的提升

- 从“会看”升级到“会想”
- 从“会想”升级到“会做”
- 从“会做”升级到“会复盘”
- 从“会复盘”升级到“越来越懂每个租户”

## 3.5 第一段已落地骨架（2026-03-07）

当前已完成的不是“真实接入 LiteLLM/OpenLIT”，而是先把独立编排边界落下：

- `IntelligenceModelGatewayOrchestrator`
- `IntelligenceObservabilityOrchestrator`
- `IntelligenceBrainSnapshotResponse.modelGateway`
- `IntelligenceBrainSnapshotResponse.observability`

当前策略：
- 默认关闭
- 只暴露状态
- 不改现有 AI 顾问和现有 intelligence 主链
- 不影响生产、库存、结算、系统设置的既有业务事务

这样做的意义：
- 先立边界，再接真实网关和真实观测
- 防止模型网关、观测埋点、供应商适配逻辑污染业务 Orchestrator
- 为下一阶段灰度接入 LiteLLM / OpenLIT / Langfuse 提供稳定挂点

### 当前已进一步落地的第二步

在上述骨架基础上，已经补上真实神经链的第一条主通道：

- `IntelligenceInferenceOrchestrator`
- `IntelligenceInferenceResult`
- `AiAdvisorService -> IntelligenceInferenceOrchestrator`
- `IntelligenceObservabilityOrchestrator.recordInvocation()`

当前行为：
- 优先走 LiteLLM 网关
- 网关失败且允许回退时，自动回退直连模型
- 每次调用统一产出 provider / model / latency / fallback / error 元数据
- 若启用 AI 观测，则统一记一条可追踪观测日志

这一步的战略意义：
- 大脑从“有状态骨架”升级成“有真实推理主干”
- 后续所有需要 LLM 的模块都应优先复用这条独立推理链，而不是各自散写 HTTP 调用
- 为后面接动作中心、案例召回、智能日报、自然语言深度解释提供统一神经总线

### 当前已进一步落地的第三步

在统一推理主干之上，已经开始把“看见问题”真正转成“下一步动作”：

- `ActionCenterOrchestrator`
- `FollowupTaskOrchestrator`
- `SmartEscalationOrchestrator`
- `GET /api/intelligence/action-center`
- `IntelligenceBrainOrchestrator -> ActionCenterOrchestrator`

当前行为：
- 统一聚合交付风险、工厂停滞、异常告警、待处理通知、财务审核发现
- 统一转换为标准动作任务结构，而不是各域各自拼接建议文案
- 统一给出升级等级与时效提示，形成 L1/L2/L3 动作分层
- 大脑快照中的动作建议已改为委托动作中心生成

这一步的战略意义：
- 大脑从“会判断”正式进入“会派发下一步动作”的阶段
- 生产域与财务域开始共用同一套动作神经，后续更容易落待办、审批、执行、回写学习
- 为后面接入 Temporal 或本地持久任务表建立了稳定动作边界

## 4. 全系统业务域智能化地图

## 4.1 开发端智能化

目标：让开发端从“录资料”升级为“会提醒、会补全、会预估”的引导系统。

能力清单：
- 款式资料完整度评分
- BOM 缺失和异常用量识别
- 工序模板智能建议
- 相似款报价区间建议
- 打版阶段下一步建议
- 样衣风险提示
- 供应商候选推荐
- 历史同款缺陷经验召回

PC 端动作：
- 新增 StyleSituationCard
- 新增 QuoteReasonCard
- 新增 BlockingFieldGuide

小程序端动作：
- 研发/管理小程序页面补“缺项提示”和“下一步引导”

建议编排器：
- StyleSituationOrchestrator
- StyleCompletenessOrchestrator
- StyleQuoteReasonOrchestrator

## 4.2 生产端智能化

目标：让生产不只是看进度，而是让系统自己发现风险、卡点和最优动作。

能力清单：
- 扫码前智能预检
- 订单交付风险评分
- 工序卡点识别
- 工厂负载识别
- 智能派工推荐
- 动态防重复间隔
- 停滞预警升级机制
- 节点完工偏差解释
- 订单根因分析
- 订单下一步建议

PC 端动作：
- 订单列表增加活体风险列
- 订单详情增加“系统判断区”和“系统建议区”
- 节点弹窗增加“为什么这样判定”与“建议处理动作”

小程序端动作：
- 扫码前预检反馈

## 6. 开源技术栈建议

目标：尽量使用免费、开源、可自托管、可二开的底座。

## 6.1 推荐作为核心底座的组件

### A. 模型运行层
- Ollama

定位：
- 本地/内网部署开源模型
- 生成解释、建议、总结、根因分析说明

建议用途：
- AI 顾问
- 建议解释器
- 根因分析摘要
- 工厂端辅助问答

### B. 向量记忆层
- Qdrant

定位：
- 案例召回
- 工艺知识召回
- 相似订单、相似缺陷、相似款式召回

建议用途：
- 交付风险历史案例回查
- 报价建议参考召回
- 质量缺陷经验召回

### C. 智能可观测层
- Langfuse

定位：
- 智能建议链路追踪
- Prompt 版本管理
- 结果评估
- 用户反馈闭环

建议用途：
- 跟踪 AI 顾问回答质量
- 跟踪建议命中率与误报率
- 追踪模型版本与策略版本变化

## 6.2 不建议作为核心依赖的组件

- n8n：适合自动化编排，但许可证不是纯开源路线下的最优核心依赖
- Dify：适合搭 AI 应用平台，但更适合作为外围实验平台，不建议压在核心供应链底层

结论：
- 核心业务主脑继续放在现有 Spring Boot intelligence 模块内
- 外围能力用 Ollama + Qdrant + Langfuse 做增强

## 11. 成功标准

这套系统是否“活起来”，不看模型名字，看以下指标：

1. 关键异常发现提前量
- 风险是否在用户人工发现前就被系统提出

2. 建议采纳率
- 用户是否愿意按系统建议执行

3. 误报率
- 智能建议是否足够准，而不是扰民

4. 动作闭环率
- 风险提出后，是否真的被人跟进和解决

5. 偏差收敛速度
- 预测偏差、建议失误是否随时间下降

6. 租户满意度
- 不同租户是否觉得建议更像“懂自己”的助手，而不是统一模板

## 12. 结论

这套系统完全可以升级成双端全链路的智能中枢，但关键不在于“再加几个 AI 功能”，而在于把现有智能能力重组为：

- 统一感知
- 统一判断
- 统一行动
- 统一记忆
- 统一学习

最终形态不是一个会聊天的页面，而是一套能主动发现问题、主动给出动作、主动积累经验、主动适配租户的供应链运营神经系统。
