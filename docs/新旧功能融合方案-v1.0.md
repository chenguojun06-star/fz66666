## 2. 功能兼容性评估

### 2.1 评估维度与矩阵

对每个新旧功能交叉点，从以下5个维度进行兼容性评估:

| 维度 | 说明 | 评分(1-5) |
|------|------|-----------|
| **接口兼容** | 新旧接口签名/返回结构是否一致 | 5=完全兼容, 1=破坏性变更 |
| **数据兼容** | 新旧数据模型是否可共存 | 5=完全共存, 1=需迁移 |
| **行为兼容** | 新逻辑是否改变旧功能行为 | 5=无影响, 1=行为变更 |
| **配置兼容** | 新配置是否影响旧配置 | 5=独立配置, 1=配置冲突 |
| **部署兼容** | 新功能是否影响旧功能部署 | 5=独立部署, 1=需同部署 |

### 2.2 核心功能兼容性评估结果

#### 2.2.1 AI推理双轨 — 风险等级: 🟡 中

| 维度 | 评分 | 说明 |
|------|------|------|
| 接口兼容 | 5 | Spring AI与自研引擎完全隔离，Bean名称带springAi前缀 |
| 数据兼容 | 5 | 两套配置源完全独立(`ai.deepseek.*` vs `spring-ai.adapter.*`) |
| 行为兼容 | 5 | Spring AI适配层无消费者，不影响现有AI调用链路 |
| 配置兼容 | 4 | `@ConditionalOnProperty`控制激活，但需注意Bean冲突 |
| 部署兼容 | 4 | Spring AI starter为optional依赖，需确认运行时classpath |

**兼容性结论**: 当前完全兼容。Spring AI处于"已定义但未消费"状态，激活需创建桥接层。

**关键约束**: Spring AI的ChatClient不得直接替换IntelligenceInferenceOrchestrator，必须通过适配器渐进替换。

| 维度 | 评分 | 说明 |

| 数据兼容 | 5 | 桥接层无持久化，纯内存注册表 |
| 行为兼容 | 5 | 内部工具调用路径不变，MCP仅增加外部调用入口 |
| 配置兼容 | 5 | 无额外配置依赖 |
| 部署兼容 | 5 | 独立Service，无部署耦合 |

**兼容性结论**: 完全兼容。MCP桥接是纯增量设计，不侵入现有Agent执行链路。

#### 2.2.3 闭环学习(OrderLearning) — 风险等级: 🟡 中

| 维度 | 评分 | 说明 |
|------|------|------|
| 接口兼容 | 4 | 新增Controller端点，不修改旧接口 |

### 3.2 统一AI推理接口设计

#### 3.2.1 AiInferenceGateway — 统一推理网关

#### 3.2.2 AiInferenceRequest — 统一请求结构

```java
@Data
@Builder
public class AiInferenceRequest {
    private String scene;
    private String systemPrompt;
    private String userMessage;
    private List<AiMessage> messages;
    private List<AiTool> tools;
    private Double temperature;
    private Integer maxTokens;
    private Long timeoutMs;
    private Map<String, String> metadata;
}
```

#### 3.2.3 适配器实现

| 适配器 | 委托目标 | 优先级 |
|--------|---------|--------|
| `LegacyInferenceAdapter` | IntelligenceInferenceOrchestrator | 主力(当前) |
| `SpringAiInferenceAdapter` | Spring AI ChatClient | 备用(渐进替换) |

**路由策略**: 通过 `ai.gateway.routing-strategy` 配置决定:
- `legacy`: 全部走自研引擎(当前默认)
- `spring-ai`: 全部走Spring AI
- `weighted`: 按权重分流(灰度场景)
- `failover`: 自研引擎优先，失败降级到Spring AI

#### 3.2.4 SpringAiInferenceAdapter 核心实现

```java
@Component
@ConditionalOnProperty(name = "spring-ai.adapter.enabled", havingValue = "true")
public class SpringAiInferenceAdapter implements AiInferenceGateway {

    @Autowired
    private ChatClient springAiChatClient;

    @Override
    public IntelligenceInferenceResult chat(AiInferenceRequest request) {
        ChatResponse response = springAiChatClient.prompt()
            .system(request.getSystemPrompt())
            .user(request.getUserMessage())
            .options(OpenAiChatOptions.builder()
                .temperature(request.getTemperature() != null ? request.getTemperature() : 0.3)
                .maxTokens(request.getMaxTokens() != null ? request.getMaxTokens() : 2048)
                .build())
            .call()
            .chatResponse();

        return convertToInferenceResult(response, "spring-ai");
    }

    @Override
    public String getProviderName() {
        return "spring-ai";
    }
}
```

## 4. 数据迁移策略

### 4.1 迁移原则

1. **增量优先**: 新增表/字段，不修改旧表结构
2. **向后兼容**: 旧字段保留，新字段默认值填充
3. **幂等执行**: 所有迁移脚本支持重复执行
4. **租户隔离**: 迁移数据必须包含tenant_id
5. **可回滚**: 每个迁移步骤有对应的回滚脚本

### 4.2 Flyway迁移规范

#### 4.2.1 命名规范

```
V{版本号}__{描述}.sql          → 正向迁移
U{版本号}__{描述}.sql          → 回滚迁移(可选)
R__{描述}.sql                  → 可重复执行迁移
```

版本号格式: `V{YYYYMMDD}{序号}`，如 `V202604301001`

#### 4.2.2 迁移脚本模板

```sql
-- 正向迁移: V{版本号}__{描述}.sql

-- 安全检查: 表不存在时才创建
CREATE TABLE IF NOT EXISTS t_new_table (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    tenant_id BIGINT NOT NULL,
    -- ... 业务字段
    create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    update_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    creator_id VARCHAR(64),
    creator_name VARCHAR(64),
    version INT NOT NULL DEFAULT 0,

    INDEX idx_{table}_tenant (tenant_id),
    INDEX idx_{table}_tenant_created (tenant_id, create_time)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

-- 数据迁移(如需从旧表迁移数据)
-- INSERT INTO t_new_table (tenant_id, ...) SELECT tenant_id, ... FROM t_old_table WHERE ...;
```

#### 4.2.3 回滚脚本模板

```sql
-- 回滚迁移: U{版本号}__{描述}.sql

-- 仅删除新增的表/字段，不恢复数据
DROP TABLE IF EXISTS t_new_table;
-- ALTER TABLE t_existing_table DROP COLUMN IF EXISTS new_column;
```

## 6. 回滚机制

### 6.1 回滚原则

1. **最小影响**: 回滚仅影响出问题的功能，不影响其他正常运行的功能
2. **快速恢复**: 回滚操作在5分钟内完成
3. **数据安全**: 回滚不丢失业务数据
4. **可观测**: 回滚操作有完整日志记录

### 6.2 功能级回滚

#### 6.2.1 功能开关回滚

通过租户智能功能开关实现即时回滚:

```yaml
# 关闭特定智能功能
smart.guide.enabled: false
smart.production.precheck.enabled: false
smart.warehousing.audit.enabled: false
```

**回滚时间**: < 1秒(数据库更新 + 5分钟缓存刷新)

**强制刷新**: 调用 `POST /api/system/tenant-smart-feature/refresh` 立即生效

#### 6.2.2 AI推理路由回滚

通过配置切换推理引擎:

```yaml
# 全部切回自研引擎
ai:
  gateway:
    routing-strategy: legacy

# 或按租户回滚
ai:
  gateway:
    routing-strategy: weighted
    tenant-overrides:
      tenant-123: legacy    # 特定租户强制走自研引擎
```

**回滚时间**: < 30秒(配置热更新)

#### 6.2.3 Spring AI模块回滚

```yaml
# 完全禁用Spring AI适配层
spring-ai:
  adapter:
    enabled: false
```

**回滚时间**: < 1分钟(需重启应用)

### 6.3 代码级回滚

#### 6.3.1 Git回滚策略

| 场景 | 回滚方式 | 命令 |
|------|---------|------|
| 单个提交有问题 | revert特定提交 | `git revert <commit-hash>` |
| 整个功能分支有问题 | 删除分支，回退到合并前 | `git revert -m 1 <merge-commit>` |
| 紧急回滚 | 回退到上一个稳定tag | `git checkout <stable-tag>` |

#### 6.3.2 分支保护规则

- `main` 分支: 仅通过PR合并，必须通过CI
- `release/*` 分支: 仅从main创建，禁止直接push
- `feature/*` 分支: 开发分支，合并后删除

### 6.4 数据库回滚

#### 6.4.1 Flyway回滚

```bash
# 查看当前迁移状态
mvn flyway:info

# 回滚到指定版本
mvn flyway:undo -Dflyway.targetVersion=202604301000
```

**前提**: 需要编写对应的U(undo)迁移脚本

#### 6.4.2 新增表回滚(安全)

新增表的回滚是安全的——直接DROP TABLE即可，不影响旧有数据:

```sql
DROP TABLE IF EXISTS t_order_decision_snapshot;
DROP TABLE IF EXISTS t_order_fulfillment_outcome;
DROP TABLE IF EXISTS t_scan_precheck_feedback;
DROP TABLE IF EXISTS t_order_risk_tracking;
DROP TABLE IF EXISTS t_ai_cost_tracking;
DROP TABLE IF EXISTS t_ai_operation_audit;
DROP TABLE IF EXISTS t_collaboration_task;
```

#### 6.4.3 数据回填回滚

如果数据回填有问题，仅删除回填的数据(按create_time区分):

```sql
DELETE FROM t_order_decision_snapshot
WHERE create_time >= '2026-04-30 00:00:00'
  AND creator_id = 'migration-script';
```

### 6.5 部署级回滚

#### 6.5.1 Docker回滚

```bash
# 回滚到上一个镜像版本

docker tag fashion-supplychain:latest fashion-supplychain:rollback
docker tag fashion-supplychain:previous fashion-supplychain:latest

```

**回滚时间**: < 30秒(仅切换Nginx/负载均衡器指向)

### 6.6 回滚决策矩阵

| 影响范围 | 严重程度 | 回滚方式 | 决策人 |

### 9.2 高风险项详细缓解方案

#### R07: 多租户数据隔离不完整

**根因分析**: 新增的7张表需要确保:
1. Entity类包含`tenantId`字段
2. `@TableField(fill = FieldFill.INSERT)` 自动填充
3. TenantInterceptor的EXCLUDED_TABLES不包含新表
4. SafeQueryHelper用于所有按ID查询

**缓解措施**:

```java
// 1. Entity必须包含tenantId字段
@Data
@TableName("t_order_decision_snapshot")
public class OrderDecisionSnapshot {
    @TableField(fill = FieldFill.INSERT)
    private Long tenantId;
    // ...
}

// 2. 所有查询必须走SafeQueryHelper
OrderDecisionSnapshot entity = SafeQueryHelper.getById(
    service, id, OrderDecisionSnapshot::getId,
    OrderDecisionSnapshot::getTenantId, "决策快照"
);

// 3. 批量操作必须校验同租户
TenantAssert.assertSameTenant(tenantIds, "批量查询决策快照");
```

**验证方法**: 编写租户隔离专项测试，模拟租户A查询租户B的数据，预期返回空。

#### R06: 闭环功能异常影响主业务流程

**缓解措施**: 所有闭环功能采用"旁路写入"模式:

```java
// 订单创建主流程
@Transactional(rollbackFor = Exception.class)
public Result<?> createOrder(OrderDTO dto) {
    // ... 主流程逻辑 ...

    // 旁路: 决策快照捕获(不影响主流程)
    try {
        orderDecisionCaptureOrchestrator.capture(dto, createdOrder);
    } catch (Exception e) {
        log.warn("决策快照捕获失败，不影响订单创建: {}", e.getMessage());
    }

    return Result.success(createdOrder);
}
```

**验证方法**: Mock闭环Orchestrator抛出异常，验证主流程正常完成。
