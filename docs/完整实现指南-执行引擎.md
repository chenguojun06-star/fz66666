## 1. 架构概述

### 问题背景

系统原先有 41 个智能编排器用于**智能判断和分析**，但无法实际执行建议。例如：
- SmartNotification 编排器输出："建议暂停订单 PO001"
- 显示在仪表板上
- 用户需要 3-5 分钟手动点击按钮执行
- 结果：AI 的建议效率不高，无法实时响应

### 解决方案

**三层执行架构**使 AI 能够从"建议"完整升级到"自动执行"：

```
┌─────────────────────────────────────────────────────────┐
│ 第1层：命令生成                                            │
│ SmartNotification → CommandGeneratorOrchestrator         │
│ 输入：AI 推荐文本                                          │
│ 输出：ExecutableCommand 结构化命令                         │
│ 职责：模式匹配、参数提取、预设风险等级                      │
└───────────────────┬─────────────────────────────────────┘
                    │
┌───────────────────▼─────────────────────────────────────┐
│ 第2层：权限决策                                            │
│ PermissionDecisionOrchestrator                          │
│ 输入：ExecutableCommand + 当前用户                        │
│ 输出：ExecutionDecision (AUTO / REQUIRES_APPROVAL / DENY)│
│ 职责：RBAC + 风险评估 + 权限检查                            │
└───────────────────┬─────────────────────────────────────┘
                    │
     ┌──────────────▼──────────────┐
     │                              │
     ▼                              ▼
  自动执行                      需要审批
  (低风险)                      (高风险)
     │                              │
     │       ┌───────────────────────┘
     │       │
     ▼       ▼
┌─────────────────────────────────────────────────────────┐
│ 第3层：执行引擎                                            │
│ ExecutionEngineOrchestrator                             │
│ 职责：                                                    │
│   1. 实际修改数据库                                        │
│   2. 记录审计轨迹                                          │
│   3. 触发工作流级联                                        │
│   4. 发送通知                                            │
│ @Transactional 保证原子性                                │
└─────────────────────────────────────────────────────────┘
```

## 2. 核心概念

### 2.1 ExecutableCommand（可执行命令）

**定义**：结构化的、可被系统理解和执行的任务指令

**属性**：
```java
{
  "commandId": "CMD-20260308-001",        // 唯一标识
  "action": "order:hold",                 // 命令类型
  "targetId": "PO202603001",              // 目标对象
  "params": {                             // 执行参数
    "holdReason": "面料延期"
  },
  "riskLevel": 3,                         // 风险等级 (1-5)
  "requiresApproval": true,               // 是否需要审批
  "reason": "面料预期延期7天，建议暂停生产", // AI建议理由
  "tenantId": 123,                        // 租户ID（多租户隔离）
  "createdAt": "2026-03-08T10:00:00",    // 创建时间
  "expiresAt": "2026-03-10T10:00:00"     // 过期时间（防止陈旧命令）
}
```

**生命周期**：
1. SmartNotification 生成建议 → CommandGenerator 转换成 ExecutableCommand
2. 命令进入待审批队列或直接执行
3. 命令在 48 小时后过期（可配置）

### 2.2 ExecutionDecision（执行决策）

**定义**：系统对该命令是否应该立即执行的决策

**三种结果**：

| 决策 | 含义 | 场景 |
|------|------|------|
| `AUTO_EXECUTE` | 低风险，直接执行 | 库存检查（风险1）+ 用户有权限 → 3秒内完成 |
| `REQUIRES_APPROVAL` | 高风险，需要人工审批 | 暂停订单（风险3）+ 风险阈值2 → 等待生产经理点击 |
| `DENIED` | 权限不够或命令过期 | 用户角色没有权限 → 返回 403 |

**决策链**：
```
命令已过期?   → YES → DENIED
    ↓NO
用户存在?    → NO → DENIED
    ↓YES
租户启用执行? → NO → REQUIRES_APPROVAL
    ↓YES
用户有权限?   → NO → DENIED
    ↓YES
风险 ≤ 阈值?  → YES → AUTO_EXECUTE
    ↓NO
             → REQUIRES_APPROVAL
```

### 2.3 ExecutionResult（执行结果）

**定义**：命令执行完成后的结果封装

**内容**：
```json
{
  "success": true,
  "data": {
    "orderId": "PO001",
    "status": "HOLD",
    "holdAt": "2026-03-08T10:05:00"
  },
  "auditId": "AUDIT-20260308-001",
  "executorId": "user_123",
  "executedAt": "2026-03-08T10:05:00",
  "cascadedTasksCount": 3,        // 触发了3个级联任务
  "notifiedRecipients": [
    "warehouse_team",
    "finance_team",
    "production_coordinator"
  ]
}
```

## 3. 3 层架构深度解析

### 第1层：CommandGeneratorOrchestrator

**输入**：SmartNotification（文本建议）
**输出**：List<ExecutableCommand>（结构化命令）

**核心工作**：
1. 文本模式匹配（正则表达式 + NLP）
2. 参数提取（从文本中识别订单号、原因等）
3. 命令类型判断（7种模式）
4. 风险等级预设

**7 种命令模式**：

| # | 模式 | 触发条件 | 风险 | 示例文本 |
|----|------|---------|------|----------|
| 1 | `order:hold` | "建议暂停" | 3 | "建议暂停订单 PO001 因为面料延期" |
| 2 | `order:expedite` | "加急"/"紧急" | 4 | "建议加急订单 PO002 避免交期逾期" |
| 3 | `order:resume` | "恢复"/"继续" | 2 | "订单已补齐物料，建议恢复生产" |
| 4 | `purchase:create` | "采购"/"下单" | 4 | "缺少面料，建议立即采购 500m" |
| 5 | `inventory:check` | "检查库存" | 1 | "核查当前库存是否足够" |
| 6 | `quality:upgrade` | "升级质检" | 2 | "订单风险高，升级为100%检验" |
| 7 | `finance:review` | "财务审分" | 4 | "流程异常，需财务审批" |

**实现位置**：
```
backend/src/main/java/.../intelligence/orchestration/CommandGeneratorOrchestrator.java
主方法：generateCommands(SmartNotification) → List<ExecutableCommand>
```

### 第2层：PermissionDecisionOrchestrator

**输入**：ExecutableCommand + userId
**输出**：ExecutionDecision
**职责**：权限检查 + 风险评估

**RBAC 权限映射**：

```
command type        required role              comment
────────────────────────────────────────────────────────
order:hold         PRODUCTION_MANAGER          管理员级权限
order:expedite     PRODUCTION_MANAGER          管理员级权限
order:resume       PRODUCTION_MANAGER          管理员级权限
purchase:create    PROCUREMENT_MANAGER         采购权限
inventory:check    WAREHOUSE_STAFF             仓库员权限
quality:upgrade    QUALITY_MANAGER             质检权限
finance:review     FINANCE_MANAGER             财务权限
```

**风险阈值**：
- 租户可配置 `autoExecutionThreshold` (默认=2)
- 风险≤阈值 → 自动执行（如无权限冲突）
- 风险>阈值 → 需要人工审批

**流程图**：
```
用户请求执行命令
    ↓
检查命令是否过期
  ├─ 过期 → DENIED
  └─ 未过期 ↓
检查用户是否存在
  ├─ 不存在 → DENIED
  └─ 存在 ↓
检查租户是否启用自动执行
  ├─ 未启用 → REQUIRES_APPROVAL
  └─ 已启用 ↓
获取用户角色列表
检查用户是否有命令所需的角色
  ├─ 无 → DENIED
  └─ 有 ↓
对比 command.riskLevel 与 tenantConfig.autoExecutionThreshold
  ├─ riskLevel ≤ threshold → AUTO_EXECUTE
  └─ riskLevel > threshold → REQUIRES_APPROVAL
```

**实现位置**：
```
backend/src/main/java/.../intelligence/orchestration/PermissionDecisionOrchestrator.java
主方法：decide(ExecutableCommand, Long userId) → ExecutionDecision
```

### 第3层：ExecutionEngineOrchestrator

**最关键**：这是真正修改数据库的层

**核心特点**：
- `@Transactional(rollbackFor = Exception.class)` 保证原子性
- 所有修改操作在一个事务内完成
- 任何错误都会导致全部回滚

**7 个执行方法**：

```java
1. executeOrderHold()
   - 修改订单状态: HOLD
   - 设置: holdReason, holdBy, holdAt
   
2. executeOrderExpedite()
   - 设置: expedited=true, expeditedAt, expeditedBy
   - 更新优先级: normal → high
   
3. executeOrderResume()
   - 恢复状态: HOLD → IN_PROGRESS
   - 清空: holdReason, holdBy, holdAt
   
4. executePurchaseCreate()
   - 调用 purchaseService 创建采购单
   - 返回: 新采购单号
   
5. executeInventoryCheck()
   - 查询库存表
   - 返回: 当前库存状态
   
6. executeQualityUpgrade()
   - 设置: qualityLevel="strict"
   - 设置: inspectionRate=100%
   
7. executeFinanceReview()
   - 创建财务审核任务
   - 返回: 审核任务ID
```

**实现位置**：
```
backend/src/main/java/.../intelligence/orchestration/ExecutionEngineOrchestrator.java
主方法：execute(ExecutableCommand, Long executorId) → ExecutionResult
  
执行流程：
  1. recordAuditStart()
  2. dispatch(command.action) → executeXxx()
  3. triggerPostExecutionWorkflow()
  4. recordAuditSuccess()
  5. return ExecutionResult
```

## 5. 风险级别和权限映射

### 风险等级定义

| 等级 | 名称 | 自动执行？ | 示例命令 | 审批等待 |
|------|------|----------|---------|----------|
| 1 | 低风险 | ✅ 是 | 库存检查 | 0 秒 |
| 2 | 低风险 | ✅ 是* | 订单恢复、质检升级 | 0 秒 |
| 3 | 中风险 | ❌ 否 | 订单暂停 | 需审批 |
| 4 | 高风险 | ❌ 否 | 创建采购、财务审分 | 需审批 |
| 5 | 极高风险 | ❌ 否 | 订单取消（未实装） | 需审批 |

*默认阈值=2，租户可调整

### 权限要求

```
order:hold (风险3)
  ├─ 需要角色: PRODUCTION_MANAGER
  ├─ 拥有角色: ROLE_SUPER_ADMIN → AUTO_EXECUTE
  ├─ 拥有角色: ROLE_production_manager → 需审批（风险>阈值）
  └─ 其他: DENIED

purchase:create (风险4)
  ├─ 需要角色: PROCUREMENT_MANAGER
  ├─ 拥有角色: ROLE_procurement_manager → 需审批
  └─ 其他: DENIED

inventory:check (风险1)
  ├─ 需要角色: WAREHOUSE_STAFF
  ├─ 拥有角色: ROLE_SUPER_ADMIN → AUTO_EXECUTE
  ├─ 拥有角色: ROLE_warehouse_staff → AUTO_EXECUTE
  └─ 其他: DENIED
```

## 7. 使用示例

### 场景 1：自动处理库存检查

```
AI SmartNotification:
  "订单 PO001 缺少100米棉布，建议检查是否有库存可替代"

流程：
  1. CommandGenerator 生成 → action="inventory:check", risk=1, requiresApproval=false
  2. PermissionDecision → 检查用户是否有权限 → 有 → AUTO_EXECUTE
  3. ExecutionEngine → 执行 executeInventoryCheck() → 查询库存表
  4. 返回: "当前库存充足，有1500米备用料"
  5. AuditTrail 记录执行日志
  6. SmartWorkflow 触发: 无需级联

耗时: < 1 秒
```

### 场景 2：需要审批的订单暂停

```
AI SmartNotification:
  "订单 PO002 面料预期延期7天，建议暂停生产避免成本浪费"

流程：
  1. CommandGenerator 生成 → action="order:hold", risk=3, requiresApproval=true
  2. PermissionDecision → 检查用户角色 → 有权限但风险高 → REQUIRES_APPROVAL
  3. 返回前端: "该命令需要生产经理审批"
  4. 前端显示待审批列表（AiExecutionPanel）
  5. 用户(生产经理)点击[执行]按钮
  6. ExecutionEngine → 执行 executeOrderHold()
     - 修改订单状态为 HOLD
     - 通知财务部
     - 通知生产协调员
  7. SmartWorkflow 触发级联:
     - 创建库存清点任务
     - 创建财务评估任务
  8. AuditTrail 记录完整操作链

耗时: 用户点击到完成 < 3 秒
```

### 场景 3：多步工作流级联

```
用户执行: order:hold (订单暂停)

ExecutionEngine 执行:
  ├─ updateOrderStatus(HOLD)
  ├─ auditTrail.logSuccess()
  └─ smartWorkflow.generatePostExecutionWorkflow()
      ├─ createFollowupTask("inventory:check")
      ├─ notifyTeam("finance", "评估成本")
      ├─ notifyTeam("production", "暂停工序")
      └─ notifyKpiDashboard("order_paused", "high")

结果: 用户一个操作,系统自动触发 4 个后续任务

通知:
  - 仓库员: "请清点订单 PO002 的物料"
  - 财务: "订单 PO002 已暂停,请评估已支付成本"
  - 生产: "订单 PO002 已暂停,所有工序请停止推进"
  - 高管: "⚠️ 订单 PO002 因面料延期已暂停"
```

## 8. 工作流级联规则

### 什么是级联？

"一个命令执行后,自动触发相关的后续任务"。避免人工逐项跟进。

**执行前**（传统流程）：
```
用户点击"暂停" → 
  手动通知仓库清点 (5分钟) →
  手动通知财务评估 (10分钟) →
  手动标记生产暂停 (5分钟) →
总耗时: 20+ 分钟，容易遗漏
```

**执行后（级联）**：
```
用户点击"暂停" → 
  自动触发 4 个任务 (3 秒) →
  各团队同时收到通知 →
  实时状态更新
总耗时: < 3 秒，100% 完整
```

### 级联规则表

| 命令 | 触发的后续任务 | 通知的团队 |
|------|---------------|----------|
| **order:hold** | 库存清点、成本评估 | 仓库、财务、生产...(4个) |
| **order:expedite** | 优先级升级、通知工人 | 工人、采购、生产...(2个) |
| **purchase:create** | 询价、合同草稿、预算 | 采购、财务、法务...(3个) |
| **quality:upgrade** | 质检计划、通知工厂 | 质检、工厂、订单方...(2个) |
| **finance:review** | 风险审评、费用审批 | 财务总监、CFO...(2个) |

### 配置级联规则

**位置**：`t_intelligence_execution_config` 表

```sql
UPDATE t_intelligence_execution_config
SET max_cascade_depth = 3,      -- 最多触发3层级联
    cascade_enabled = true      -- 启用级联
WHERE tenant_id = 123;
```

**代码**：`SmartWorkflowOrchestrator.java`

## 9. 审计和监控

### 审计日志表

**表名**：`t_intelligence_audit_log`

**核心字段**：

| 字段 | 含义 | 示例 |
|------|------|------|
| `command_id` | 命令唯一ID | CMD-20260308-001 |
| `action` | 命令类型 | order:hold |
| `executor_id` | 执行人 | user_123 |
| `status` | 执行结果 | SUCCESS/FAILED |
| `duration_ms` | 耗时 | 2340 |
| `result_data` | 执行结果(JSON) | {...} |
| `error_message` | 错误信息 | 权限不足 |
| `created_at` | 创建时间 | 2026-03-08 10:00:00 |

## 10. 故障排查

### 问题 1：命令返回 403 DENIED

**症状**：
```json
{
  "code": 403,
  "message": "权限不足"
}
```

**解决**：
1. 检查用户是否拥有所需角色
2. 检查 PermissionDecisionOrchestrator 中的角色映射
3. 运行: `SELECT roles FROM t_user WHERE id='user123';`

### 问题 2：命令执行失败，但找不到错误

**症状**：
```json
{
  "success": false,
  "errorMessage": "未知错误"
}
```

**解决**：
1. 查询审计日志: 
   ```sql
   SELECT * FROM t_intelligence_audit_log 
   WHERE command_id='CMD-xxx' AND status='FAILED';
   ```
2. 查看 `error_message` 字段
3. 检查后端日志: `tail -f logs/fashion-supplychain.log | grep ExecutionEngine`

### 问题 3：级联任务没有触发

**症状**：命令执行成功，但 `cascadedTasksCount=0`

**解决**：
1. 检查 SmartWorkflowOrchestrator 中的级联规则
2. 检查该命令类型是否有对应的规则
3. 查询工作流日志:
   ```sql
   SELECT * FROM t_intelligence_workflow_log 
   WHERE command_id='CMD-xxx';
   ```

### 问题 4：批量命令执行时变慢

**症状**：10 个命令执行耗时 >30 秒

**解决**：
1. 检查数据库连接数: `SHOW PROCESSLIST;`
2. 检查是否有慢查询: 审计日志查询，库存查询等
3. 添加索引: `CREATE INDEX idx_order_id ON t_production_order(order_id);`
