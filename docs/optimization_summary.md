## 二、核心优化内容

### 2.1 多端类型统一（关键修复）

**问题：**
- PC端、小程序、H5端使用不同的类型定义
- 业务逻辑修改需要同时更新3处
- 容易出现类型不一致导致的Bug

**解决方案：**
- 创建统一的共享类型定义库
- TypeScript 严格类型检查
- 自动生成工具

**新增文件：**


### 2.2 缓存优化与数据一致性

**现有基础：**
- Redis 已配置
- 已有本地缓存（Caffeine）
- 缺少统一的缓存管理

**优化方案：**
- 统一的缓存管理器
- 自动的缓存失效
- 实时的缓存同步

**新增核心类：**

| 文件 | 功能 |
|------|------|
| `UnifiedCacheManager.java` | 统一缓存管理 |
| `RealTimePushService.java` | 实时推送服务 |
| `DataSync.java` | 数据同步注解 |
| `DataSyncAspect.java` | 数据同步切面 |

**缓存流程：**

```
数据修改
    ↓
@DataSync 注解触发
    ↓
1. 失效 Redis 缓存
2. 发布 Redis 同步事件
3. WebSocket 推送到客户端
4. 客户端自动刷新数据
```

**使用示例：**

```java
@DataSync(
    entityType = "order",
    eventType = "order:update",
    cacheName = "order"
)
public ProductionOrder updateOrder(ProductionOrder order) {
    // 业务逻辑...
    return order;
}
```

### 2.4 数据库索引优化

**问题：**
- 扫码查询慢（订单查询）
- 库存查询慢
- 工资计算聚合慢
- 多租户查询慢

**优化方案：**
- 关键业务表全量索引
- 复合索引优化
- 租户索引优化
- 时间索引优化

**新增迁移文件：**
- `V202605300000__performance_indexes.sql`

**核心索引：**

| 表 | 索引 | 优化效果 |
|-----|------|---------|
| `t_production_order` | `order_no`, `factory_id`, `status`, `tenant_id` | 订单查询加速 5-10x |
| `t_scan_record` | `order_id`, `scanned_at DESC`, `tenant_id` | 扫码查询加速 10x+ |
| `t_material_stock` | `material_code`, `warehouse_id`, `tenant_id` | 库存查询加速 5x |
| `t_cutting_bundle` | `order_id`, `status`, `tenant_id` | 裁剪查询加速 5x |

**预计性能提升：**
- 查询性能提升 **50-80%**
- 扫码响应从 2-3秒 降至 **<500ms**
- 工资计算从 10秒 降至 **<2秒**
