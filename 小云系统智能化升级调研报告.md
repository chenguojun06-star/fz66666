- **工具调用级审计**：每次工具调用的参数、结果、耗时
- **降级路径追踪**：主模型→备用模型→兜底的完整路径
- **Token 碎片化分析**：提示词结构效率、冗余 content 占比
- **供应商依赖仪表盘**：各 LLM Provider 的可用性、延迟 P50/P95/P99

### 2.2 Structured Output 约束解码

**技术原理**

LLM 输出不可靠是生产事故的首要原因。Structured Output（结构化输出）通过约束 LLM 的 token 采样空间，确保输出严格符合 JSON Schema。2025 年三大技术路线：

| 路线 | 原理 | 代表方案 | 准确率 |
|------|------|---------|:------:|
| **Guided Decoding** | 解码层约束 token 选择 | vLLM + XGrammar / Outlines | 100% |
| **JSON Mode** | 模型训练时内置 | OpenAI gpt-4o JSON mode | 99.9% |
| **Retry + Validation** | 解析失败重试 | Instructor / StrictJSON | 95-98% |

**对小云的可实施性**：⭐⭐⭐⭐

小云当前 LLM 调用依赖 OpenAI 兼容接口，模型端（DeepSeek）支持 JSON Mode。小云的 75+ 工具依赖 LLM 输出 JSON 参数，输出结构化可避免「工具调用参数解析失败」类错误。

**实施路径**（1-2周）：

1. 在 `IntelligenceInferenceOrchestrator.invokeOpenAiCompatible()` 中增加 `response_format` 参数
2. 对工具调用的 function callings 强制使用 JSON mode
3. 对 NL 查询场景使用结构化输出约束 `{intent, confidence, answer}`

**预期效益**：工具调用参数解析失败率降低 80%，无需 retry→节省 Token

### 2.3 自适应超时与部分恢复

**技术原理**

`ai-fallback` 库提出的「自适应超时」：超时值不再硬编码，而是根据历史 P95 延迟 + 20% buffer 动态计算。「部分恢复」：流式输出被截断时，捕获已接收的内容而非完全丢弃。

**对小云的可实施性**：⭐⭐⭐⭐

小云的 `AbstractAgentTool.executeWithTimeout()` 当前固定 30 秒超时。升级为自适应：

```java
// 当前：固定30秒
future.get(30, TimeUnit.SECONDS);

// 升级后：P95延迟 * 1.2，下限5秒，上限60秒
int adaptiveTimeout = Math.min(60000, Math.max(5000, 
    (int) (p95Latency.rollingAverage() * 1.2)));
future.get(adaptiveTimeout, TimeUnit.MILLISECONDS);
```

**实施路径**（1周）：
1. 新增 `LatencyTracker` 组件，记录各工具 P50/P95/P99 延迟
2. `resolveTimeoutMs()` 改为自适应计算
3. 流式输出增加 truncated content 捕获逻辑

## 四、方向三：复杂任务智能化工具集成方案

### 4.1 MCP 工具协议标准化

**技术原理**

MCP（Model Context Protocol）是 Anthropic 2024 年开源的「AI 工具 USB-C」标准。截至 2026 年 Q1，9700 万次安装，4000+ MCP Servers。核心机制：
- **Tool Discovery**：Agent 自动发现 Server 提供的所有工具及参数
- **Context Sharing**：统一上下文格式，消除工具间数据格式差异
- **Gateway Pattern**：多 Server 统一接入点，OAuth 认证

2025 年 MCP 生态关键项目：
- `agentic-community/mcp-gateway-registry`：企业级 MCP Gateway + Registry
- `Microsoft/mcp`：微软官方 MCP Server 实现目录
- `MCPilotX/IntentOrch`：意图驱动的 MCP 编排

**对小云的可实施性**：⭐⭐⭐⭐⭐（P1 最高优先级）

小云 75+ 工具使用自定义 `AiTool` 接口，维护成本高，不利于生态扩展。MCP 标准化后：

```
之前：其他AI应用 → [无法接入] → 小云数据
之后：其他AI应用 → [MCP协议] → 小云MCP Server → 小云数据
```

**实施路径**（2-4周）：

1. 封装核心工具为 MCP Server：
   - `mcp-order-server`：订单查询、进度查询、延期分析
   - `mcp-scan-server`：扫码统计、工序查询、个人产量
   - `mcp-finance-server`：工资查询、对账查询、应付查询
2. 实现 MCP Gateway 统一入口
3. 保持旧 `AiTool` 接口兼容（内部调用 MCP Server），渐进迁移

**预期效益**：工具维护成本降低 50%，外部 AI 应用可直接接入小云数据

**潜在风险**：迁移期间双接口维护成本，采用渐进策略（新工具 MCP + 旧工具兼容）

### 4.2 DAG 工作流用户可视化编排

**技术原理**

LangGraph 的核心思想：将 Agent 工作流建模为有向状态图（StateGraph），节点是处理步骤，边是条件分支。支持：
- **并行执行**：无依赖节点同时执行
- **条件路由**：根据节点输出动态选择后续路径
- **人机协作节点**：需要人工确认的节点暂停并等待输入

> 注意：Temporal.io 2025 年的反思文章指出「过度使用 DAG 图」的问题，建议复杂 Agent 逻辑用代码而非图形。对于小云，应在「用户可见层」使用 DAG 可视化，「内部执行层」用代码。

**对小云的可实施性**：⭐⭐⭐⭐

小云已有 `DagExecutionEngine` + `DagGraph`，但主要用于内部 Agent 循环。可升级为用户可见的「任务可视化」：

```
订单全链路分析 DAG：

  [查询订单] ──→ [扫码统计] ──→ [进度计算]
       │                              │
       └──→ [延期检测] ──→ [根因分析] ──→ [报告生成]
                              │
                          [人工确认] ← 高风险建议需审批
```

**实施路径**（1-2月）：

1. 为 `DagGraph` 增加可视化元数据（nodeLabel、nodeIcon、progressWeight）
2. 前端实现 DAG 可视化组件（类似 GitHub Actions workflow 图）
3. `AgentLoopEngine` 将复杂任务拆分为 DAG 子任务，支持并行执行和进度追踪
4. 人工确认节点暂停 DAG 执行，等待审批后继续

### 4.3 意图驱动的工具编排（Intent-Driven Orchestration）

**技术原理**

`MCPilotX/IntentOrch` 项目提出的核心创新：用户用自然语言描述需求，系统自动将意图解析为 DAG 执行计划：

```
用户: "分析OR-001为什么延期"

IntentOrch:
  1. 意图解析 → {intent: root_cause_analysis, target: OR-001}
  2. 动态计划生成 →
     [查询订单状态]
        ├→ [检查物料到位率]
        ├→ [检查人员排班]
        ├→ [检查设备状态]
        └→ [综合报告]
  3. 执行 + 结果聚合
```

**对小云的可实施性**：⭐⭐⭐⭐

小云的 `NlQueryOrchestrator` 已有 34 个意图分类，可升级为自动生成执行计划：

**实施路径**（3-6周）：
1. 为每个已知意图定义「执行计划模板」：包含子任务列表、依赖关系、所需工具
2. `NlQueryOrchestrator` 增加 `planIntent()` 方法，将意图映射为 DAG 图
3. `DagExecutionEngine` 执行 DAG，聚合结果

### 5.3 ToT（Tree of Thoughts）树状思维探索（P1）

**技术原理**

ETH Zurich 的 Tree of Thoughts 论文提出的核心创新：将推理过程从 **线性链** 升级为 **树状结构**：

```
                    [根问题: OR-001为什么延期?]
                    /           |           \
            [物料短缺?]    [人员不足?]    [设备故障?]
              /    \         /     \        /     \
         [面料？] [辅料？] [裁剪组?] [车缝组?] [裁床?] [缝纫机?]
           ↓       ↓       ↓       ↓       ↓       ↓
         [确认: 面料延期3天导致] ← 回溯剪枝，整合结论
```

**核心算法**：

```
Algorithm ToT(problem, max_depth=3, branching_factor=3):
    root = create_thought_node(problem)
    frontier = [root]
    for depth in 1..max_depth:
        next_frontier = []
        for node in frontier:
            thoughts = generate_branches(node, branching_factor)
            evaluated = evaluate_thoughts(thoughts)
            pruned = keep_top_k(evaluated, k=branching_factor/2)
            next_frontier.extend(pruned)
        frontier = next_frontier
    return extract_best_path(root)
```

**对小云的可实施性**：⭐⭐⭐⭐

小云已有「Self-Consistency」的多路径推理、「Critic」的评估能力，ToT 是其自然进化：

| 现有能力 | ToT 映射 |
|---------|---------|
| `SelfConsistencyVerifier.sample()` | `generate_branches()` |
| `AiCriticOrchestrator.reviewAndRevise()` | `evaluate_thoughts()` |
| `AgentLoopEngine.stuckDetection` | 剪枝逻辑 |

**适用场景**：
- 订单延期根因分析（`RootCauseAnalysisOrchestrator`）：每个假设作为分支
- 产能瓶颈检测（`CapacityGapOrchestrator`）：多维度并行探索
- 智能排程建议（`SchedulingSuggestionOrchestrator`）：多条排程方案比较

**实施路径**（2-3周）：
1. 新增 `TreeOfThoughtsEngine` 组件，实现 BFS/Beam Search 搜索策略
2. 在 `RootCauseAnalysisOrchestrator` 中集成 ToT 模式
3. ToT 搜索过程通过 SSE 实时推送给前端

**资源需求**：

- 计算资源：与 Self-Consistency 共享线程池

### 5.4 GoT（Graph of Thoughts）图状思维推理（P2）

**技术原理**

ETH Zurich 2023 年论文提出，2025 年 SPCL 实验室进一步发展为 KGoT（Knowledge Graph of Thoughts）：

与 ToT 的核心区别：
- ToT：**树状结构**，节点只能有一个父节点，无法聚合多个分支
- GoT：**图状结构**，节点可有多个入边，支持「多源聚合」

```
ToT（树状）：
  A → B1 → C1            ← 每个节点单一父节点
    → B2 → C2

GoT（图状）：         
  A → B1 ──┐
    → B2 ──┼→ D（聚合节点） ← 多源汇聚
    → B3 ──┘
```

KGoT 在此基础上集成了 **动态知识图谱**：
- 每次工具调用结果存入图谱
- 后续推理可通过图遍历快速发现关联事实
- 特别适合供应链「订单→款式→BOM→物料→供应商」这种多跳关系推理

**对小云的可实施性**：⭐⭐⭐（P2，中长期）

小云的供应链数据天然是图结构：
- 实体：订单、款式、BOM、物料、供应商、工人
- 关系：生产、采购、质检、结算、派工

**实施路径**（2-3月）：
1. 构建供应链知识图谱（基于 Neo4j 或 Qdrant 图索引）
2. 在 `RootCauseAnalysisOrchestrator` 中增加 GoT 模式
3. 低风险场景（非工资/非财务）默认使用 GoT

**资源需求**：
- 知识图谱数据库：Neo4j 社区版（免费）或 Qdrant
- 开发工时：约 8-12 人天
- 维护成本：中（图谱增量更新）

### 5.5 工具集成长 CoT（Tool-Integrated Reasoning）（P1）

**技术原理**

`ChengpengLi1003/CoRT` 等项目的核心发现：让 LLM 在推理过程中**自发调用工具**，而非先推理再调用。当前小云的 Agent 循环是「推理→选择工具→执行→继续推理」，CoT 模式则是「推理中途发现需要数据→调用工具获取→继续推理」。

```
当前（工具调用在推理之后）：
  LLM推理：我需要知道OR-001的物料到位率
  → 调用 tool_material_picking [1秒]
  → LLM推理：物料到位率60%，延迟3天
  → 调用 tool_production_progress [1秒]
  → LLM推理：总延期约5天

CoRT（工具调用在推理之中）：
  LLM推理：我需要知道OR-001的物料到位率
  <调用 tool_material_picking> [1秒]
  得到结果60%... 这意味着延迟3天...
  还需要知道当前生产进度
  <调用 tool_production_progress> [1秒]
  综合来看总延期约5天
```

**对小云的可实施性**：⭐⭐⭐⭐

**实施路径**（3-4周）：
1. `AgentLoopEngine` 增加 CoRT 模式：推理阶段检测工具需求模式
2. 工具调用结果内联到推理 text 中（作为 system message 注入）
3. 对数学计算密集场景（工资计算、成本预估）自动触发代码解释器

## 十、结论与建议

### 核心结论

1. **小云已具备优秀的智能化基础设施**：Agent 循环、75+ 工具、四级权限、三级降级、审批门——这些是行业一流水平
2. **当前最优投入方向是「可靠性 × 可观测性」**：第二阶段的可观测性升级是最快见效的投资
3. **认知推理升级应循序渐进**：Self-Consistency（已做）→ ToT（P1）→ GoT+KGoT（P2），每步验证后再推进
4. **MCP 标准化是高杠杆战略投入**：一次投入，永久降低工具维护成本，并打开外部生态
5. **交互设计是「最后一公里」**：技术再强，用户体验差就无法落地。Think+Do、置信度指示、渐进式披露应同步投入

### 建议执行优先级

```
P0（已完成）：降级策略、Self-Consistency、证据增强、审批门
P0（立即）：全链路可观测性、Structured Output、渐进式披露
P1（本月）：MCP标准化、ToT引擎、工具集成长CoT
P1（下月）：意图驱动DAG编排、置信度指示
P2（季度内）：GoT+KGoT、Think+Do、DAG可视化
```
