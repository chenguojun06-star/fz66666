### 2. 代码审查要点

**业务逻辑检查**：
- [ ] 是否有重复的业务逻辑代码？→ 提取到工具函数
- [ ] 函数是否过长（>150行）？→ 拆分为小函数
- [ ] 圈复杂度是否过高（>15）？→ 简化if-else逻辑
- [ ] 是否有硬编码的业务规则？→ 提取为配置

**性能检查**：
- [ ] `setData` 是否传输大数据？→ 只传必要数据
- [ ] 是否有频繁的 `setData` 调用？→ 批量更新
- [ ] 网络请求是否有防抖节流？→ 添加防重复点击
- [ ] 列表渲染是否使用 `wx:key`？→ 提升性能

**错误处理**：
- [ ] API 调用是否有 try-catch？
- [ ] 是否有用户友好的错误提示？
- [ ] 是否有错误日志记录？

### 3. 重构优先级

**P0 - 立即重构**：
- ❌ 重复代码超过50行
- ❌ 函数复杂度 >20
- ❌ 函数行数 >200行
- ❌ 循环依赖
- ❌ 未处理的异常

**P1 - 本周重构**：
- ⚠️ 重复代码30-50行
- ⚠️ 函数复杂度 15-20
- ⚠️ 函数行数 150-200行
- ⚠️ 未使用的变量/函数
- ⚠️ 硬编码的业务规则

**P2 - 本月优化**：

## 📞 常见问题

### Q1: 工具太多，从哪开始？
**A**: 从P0工具开始：
1. 使用微信开发者工具的"代码质量"检查（无需安装）
2. 安装 Prettier 格式化代码
3. 每天运行 `npm run 检查` 和 `npm run 修复`

### Q2: ESLint 警告太多怎么办？
**A**: 分批修复：
1. 先修复 `--fix` 可以自动修复的
2. 再修复简单的（如未使用变量）
3. 最后修复复杂的（如重构）

### Q3: 如何评估代码质量？
**A**: 关注4个指标：
1. **ESLint警告数**：目标 <5个
2. **函数平均行数**：目标 <100行
3. **重复代码率**：目标 <10%
4. **圈复杂度**：目标 <15

### Q4: 性能分析发现卡顿，如何优化？
**A**: 3步排查：
1. 检查 `setData` 大小（目标 <64KB）
2. 检查 `setData` 频率（目标 <60次/秒）
3. 检查长任务（目标 <50ms）
