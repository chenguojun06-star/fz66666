## 执行摘要

本次架构审查对服装供应链管理系统进行了全面分析，识别出**3个P0级（严重）问题**、**5个P1级（重要）问题**和**若干P2/P3级改进建议**。系统整体架构合理，但在代码组织、安全细节和性能优化方面需要改进。

### 关键发现

| 类别 | 问题数量 | 最严重问题 |

| 可维护性 | 4 | 缺少DTO层、重复代码 |

## 1. 架构设计分析

### 1.1 后端分层架构

#### 当前架构

```
┌─────────────────────────────────────────────────────────────┐
│                        Controller 层                        │
├─────────────────────────────────────────────────────────────┤
│                     Orchestrator 层                         │
├─────────────────────────────────────────────────────────────┤
│                      Service 层                             │
│  - Interface + Impl 模式                                    │
│  - Query/Command 分离 (CQRS雏形)                            │
│  - Fill Service 模式                                        │
├─────────────────────────────────────────────────────────────┤
│                       Mapper/DAO 层                         │
├─────────────────────────────────────────────────────────────┤
│                       Entity 层                             │
└─────────────────────────────────────────────────────────────┘
```

#### 架构评估

**✅ 优点：**
1. **CQRS模式**：区分QueryService和CommandService
2. **Orchestrator层**：处理复杂业务流程编排
3. **Fill Service模式**：专门处理数据填充逻辑

**❌ 问题：**

| 问题 | 严重程度 | 影响 | 位置 |
|------|---------|------|------|
| Orchestrator职责过重 | 🔴 P0 | 可维护性差 | ProductionOrderOrchestrator (1658行) |
| 分层边界模糊 | 🟡 P1 | 架构混乱 | Orchestrator直接调用Mapper |
| 缺少DTO层 | 🟡 P1 | 数据暴露风险 | Controller直接返回Entity |
| 贫血模型 | 🟡 P1 | 业务逻辑分散 | 所有实体类 |

### 1.2 前端架构

#### 目录结构评估

```
frontend/src/
├── components/     ✅ 公共组件
├── modules/        ✅ 按业务域组织
├── services/       ✅ API服务层
├── stores/         ✅ 状态管理(Zustand)
├── hooks/          ✅ 自定义Hooks
├── utils/          ⚠️ 业务逻辑混入
└── types/          ✅ 类型定义
```

**✅ 优点：**
- 清晰的模块划分
- 服务层分离
- TypeScript类型定义集中

**❌ 问题：**

| 问题 | 严重程度 | 影响 | 位置 |
|------|---------|------|------|
| 组件过大 | 🔴 P0 | 维护困难 | StyleInfo/index.tsx (2711行) |
| 工具函数混杂 | 🟡 P1 | 职责不清 | api.ts包含业务逻辑 |
| 状态持久化缺失 | 🟡 P1 | 用户体验差 | Zustand无持久化配置 |

## 2. 组件间依赖关系分析

### 2.1 后端模块依赖

```
production (生产模块)
  ├── style (款式模块)        [强依赖]
  ├── finance (财务模块)      [循环依赖风险 ⚠️]
  ├── system (系统模块)       [弱依赖]
  └── warehouse (仓库模块)    [中等依赖]

finance (财务模块)
  ├── production              [强依赖]
  └── style                   [中等依赖]
```

**⚠️ 循环依赖风险：**
- production ↔ finance 相互依赖
- 通过Orchestrator层调用，增加耦合度

### 2.2 依赖注入分析

**✅ 良好实践：**
- 使用Spring的@Autowired进行依赖注入
- 构造函数注入模式（新创建的服务类）

**❌ 问题：**
- 部分Service直接依赖其他ServiceImpl而非接口
- 违反依赖倒置原则

## 3. 数据流转机制分析

### 3.1 后端数据流

```
Request → Controller → Orchestrator → Service → Mapper → Database
                                              ↓
Response ← DTO(缺失) ← Entity ← Service ← Mapper
```

**❌ 问题：**
1. **缺少DTO层**：Controller直接返回Entity
2. **Entity职责过重**：包含大量@TableField(exist = false)字段
3. **数据转换逻辑分散**：在各Service中重复实现

### 3.2 前端数据流

```
UI Component → Hook/Service → API → Backend
      ↑                            ↓
   Zustand Store ←────────────── Response
```

**✅ 优点：**
- 单向数据流
- 状态管理清晰

**❌ 问题：**
- 缺少数据缓存层
- 重复请求未优化

## 6. 安全隐患检查

### 6.1 高危安全问题

#### 🔴 P0: 文件上传路径遍历漏洞

**位置：** `ProductionOrderOrchestrator.java:611-623`

**问题代码：**
```java
String fileName = u.substring(idx + prefix.length()).trim();
if (!StringUtils.hasText(fileName) || fileName.contains("..") || fileName.contains("/")
        || fileName.contains("\\")) {
    return "";
}
```

**风险：** 黑名单验证不完整，可能被绕过

**修复建议：**
```java
// 使用UUID重命名，不依赖用户输入
String extension = FilenameUtils.getExtension(originalFilename);
String safeFileName = UUID.randomUUID().toString() + "." + extension;
```

### 6.2 中危安全问题

#### 🟡 P1: 缺少DTO层导致数据泄露

**风险：** Controller直接返回Entity，可能暴露敏感字段

**修复建议：**
```java
// 创建DTO
public class ProductionOrderDTO {
    private String id;
    private String orderNo;
    // 不包含敏感字段如costPrice等
}
```

#### 🟡 P1: API权限控制不完整

**位置：** `SecurityConfig.java`

**问题：** 部分API允许匿名访问

**建议：** 审查所有端点的权限配置
