两个实施方案对比

同一目标（精致商务风），两种落地方式。以下用「生产跟进列表」做对比——这是目前最臃肿的页面（517行）。

方案一：渐进规整（推荐）

不改页面结构，只统一视觉规范。风险最低，最快见效。

优点

  • 改动量可控 — 约 40-50 个文件，集中在 CSS + 设计系统
  • 零功能风险 — 不改组件结构，不改数据流
  • 快速验证 — 每改一个页面即可编译验证
  • 用户零学习成本 — 布局不变，只是变干净

局限

  • 治标不治本 — 超长页面（517行）的结构问题还在
  • 个别页面需补刀 — 部分页面结构本身太乱，光改CSS不够

方案二：页面模板化 + 代码瘦身

建立标准页面模板，重构核心页面结构。效果最彻底，但风险和工作量更大。

方案二 — 页面模板示意
标准页面模板 = 5 层结构
1. PageHeader — 标题 + 描述 + 操作按钮区（≤60行）
2. StatsBar — 统计卡片行（可选，≤40行）
3. FilterToolbar — 搜索+筛选+快捷标签（≤80行）
4. DataTable — 表格（列宽自适应 + ResizableTable）（≤120行）
5. EmptyState / ErrorState — 空状态统一组件（≤20行）
总行数目标：页面 index ≤ 300行（符合P0铁律）

优点

  • 根治臃肿 — 所有页面瘦身到≤300行
  • 真正统一 — 所有页面用同一套模板，不存在「各写各的」
  • 长期收益 — 新增页面自动规范

风险

  • 工作量大 — 约 80-100 个文件改动，涉及组件拆分
  • 功能回归风险 — 重构页面结构可能引入交互 bug
  • 工期长 — 需要逐页测试

推荐：方案一为主 + 方案二补刀

先用方案一统一视觉规范（CSS + 设计系统），对结构实在太乱的个别页面用方案二瘦身。

具体节奏：
第1步 → 更新 design-system.css + global.css（建立规范）
第2步 → 逐页对齐 CSS 变量 + 间距统一（不改结构）
第3步 → 对超长页面（Production List 517行 / Dashboard 423行 / StyleInfo 415行）做组件拆分瘦身
每一步都编译验证。
